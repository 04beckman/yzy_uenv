## **1\.  模块联通与 GRPO Smoke Test**



### **1\.1 测试目标**



当前目标是证明端到端链路已经连通，VeRL 能够通过 UEnv Adapter 接出 pre\-rollout episode，请求经过 Rust adapter core、Serve/Worker，再把结果回填给 VeRL，最终完成 GRPO 训练 step。



链路形态：



```Plain Text
VeRL trainer
  -> UEnvAgentLoop / Python shim
  -> Rust adapter core
  -> UEnv Server / Worker
  -> EpisodeResult
  -> VeRL AgentLoopOutput
  -> GRPO update
```



该版本重点是联通和训练流程完成，不以真实训练效果提升为主要目标。早期 Worker 侧仍包含 mock/fixture 路径，因此 v0\.0\.1 的结果主要用于说明各模块之间的接口、请求、返回和训练 step 已打通。

## **2\. Worker 替换 Mock 后的真实训练验证**



### **2\.1 测试目标**



当前目标是验证 Worker 替换 mock 数据后，链路不再只是接口联通，而是能够真实执行训练：



1. VeRL 在 pre\-rollout 阶段接出 sample。

2. Adapter 构造 prompt\-only episode 请求。

3. Rust adapter core 将请求转为 Server/Worker 所需结构。

4. Worker 使用真实 model endpoint 生成 response。

5. Worker 返回 reward、trajectory 和 response。

6. Adapter 将结果回填为 VeRL `AgentLoopOutput`。

7. VeRL 完成 GRPO 更新。

### **2\.6 结论**



当前联调证明 Worker 替换 mock 数据后，真实训练链路已经可用。VeRL 能连续完成 50 个 GRPO step，adapter 能稳定收到 Worker 返回的 response、trajectory 和 reward，并将其回填到 VeRL 训练流程。



当前 GSM8K sample 上 reward 从第一步开始就是满分，因此本次结果主要证明真实训练闭环和稳定性，不能证明模型效果继续提升。后续若要观察明显训练收益，需要使用更难的数据切片、更弱的初始模型，或提高 `ROLLOUT_N` 形成组内 reward 差异。
