## 问题 1：当前 Worker 返回的 reward 全为 0（已解决）



目前全链路已经能跑通：VeRL 在 pre\-rollout 阶段把样本交给 adapter，adapter 通过 Rust adapter core / Server 交给Worker，Worker 完成模型生成、环境 step、reward 计算后返回结果，adapter 也能收到 response\_text、trajectory 和 reward。



但目前观察到的问题是：Worker 返回的 reward 全部是 0\.0。



从现有代码和日志看，这不像是 adapter 侧覆盖了 reward，而更像是 Worker 侧数学环境判分没有通过。当前 Worker 的路径大致是：

```Plain Text
Worker ModelClient 调模型生成 action
  -> math plugin step(action)
  -> GSM8K answers_match(action, expected)
  -> 得到 plugin reward
  -> EpisodeResult.summary.total_reward 返回给 adapter
```

在 adapter 侧看到的结果里，expected 是正常的，例如 72、10，但 response\_text 经常只包含推理过程的开头，没有完整最终答案，也没有类似“72”这种 GSM8K 可解析格式。因此 answers\_match\(\) 判错后，reward 就是 0。



希望 Worker 侧协助确认几个点：



1. Worker 实际传给模型的 prompt 是什么，是否包含“最终答案用 \#\#\#\# \<answer\> 输出”的指令。

2. Worker 调模型时的 max\_tokens / max\_new\_tokens 是否太小，导致回答被截断。

3. Worker 收到的 target / expected 是否和题目匹配。

4. math plugin 判分前的原始 action 是什么，尤其是 action 的结尾部分。

5. answers\_match\(\) 提取出来的答案是什么，最终为什么判为 false。

6. 是否能在 Worker 日志中按 episode\_id 打印：question、expected、raw action tail、extracted answer、reward。



adapter这边的判断是：链路已通，但当前模型输出或 prompt 格式导致 GSM8K reward 判分失败，所以 reward 全为 0。



## 问题 2：Worker 侧有模型时，VeRL 更新后的模型如何同步（已解决）



目前训练主链路是：

```Plain Text
VeRL trainer
  -> pre-rollout 交给 adapter
  -> Server / Worker 做 rollout
  -> Worker 返回 response + reward + trajectory
  -> adapter 封装成 VeRL AgentLoopOutput
  -> VeRL 计算 advantage / loss
  -> VeRL 更新 actor 模型
```



这里有一个关键问题：如果 Worker 侧使用的是它自己维护的模型服务，那么 VeRL 更新 actor 之后，Worker 侧模型不会自动更新。



也就是说，可能出现：



第 1 轮：Worker 用模型版本 A 生成 rollout，VeRL 根据 rollout 更新出模型版本 B

第 2 轮：Worker 如果仍然用模型版本 A 生成 rollout



这样训练就不再是严格的 on\-policy GRPO。短期 smoke test 可以跑通，但长期训练效果可能不可信。

这里需要和 Worker / Server 侧明确模型所有权。



推荐方案是：



Worker 不自己维护训练模型

Worker 调用 VeRL 管理的 rollout model endpoint

VeRL 每次更新 actor 后，由 VeRL 自己负责同步权重到 rollout engine

Worker 下一轮自然调用到更新后的模型



这种情况下，Worker 的职责是：



接收 EpisodeRequest

调用指定 model\_endpoint 生成 action

执行环境 step

计算 reward

返回 EpisodeResult / trajectory



Worker 不需要计算 loss，也不需要更新模型。



如果 Worker 必须使用自己维护的模型服务，那就需要额外设计权重同步机制，例如：



1. VeRL 每 N step 保存 checkpoint，Worker reload。

2. VeRL 导出 LoRA，Worker 动态加载新 LoRA。

3. Server/Worker 增加 ReloadModel 或 UpdateWeights 接口。

4. 每个 rollout result 标记 policy\_version，保证训练样本和模型版本可追踪。

    

这部分目前还没有完全对齐。我们需要 Worker 组确认：正式训练时 Worker 调用的模型到底是谁维护的，是 VeRL rollout endpoint，还是 Worker 自己的模型服务。



我的建议是先按第一种方式推进：Worker 调 VeRL 暴露的模型 endpoint，VeRL 负责模型更新和权重同步，Worker 专注环境执行和reward。这样 adapter 的边界最清晰，也最符合当前 GRPO 训练流程。
