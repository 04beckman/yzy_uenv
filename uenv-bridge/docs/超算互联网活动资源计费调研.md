调研时间：2026-07-01

## 结论概览

该链接是“销售名片默认活动”的试用申请页，不是正式付费下单页。邀请码为 `18221793699`，活动 ID 为 `v47N2Wvl`，活动状态为 `enable`，有效试用周期字段为 `90` 天。

页面实际可选资源共有 3 个资源卡：

1. 华东一区【昆山】CPU 通用计算：`7185-32C-128G`
2. 华东一区【昆山】GPU/异构计算：`7185-32C-128G-4卡`
3. 华北一区【雄衡】CPU 通用计算：`AMD 7542-64C-256G`

从页面接口和申请页代码看，该活动按试用额度申请，不展示人民币单价；申请提交参数固定为 `type: "TRIAL"`，资源信息固定为 `SERVICE_TYPE: "SHARED"`。因此，该链接当前只能申请共享试用资源；未发现独占资源、裸金属资源或独占/裸金属切换入口。

## 数据来源

页面是前端 SPA，首屏 HTML 只加载 JS/CSS，资源与活动数据来自后端接口：

- 邀请码详情：`/acx/operation/market/invitation/detail?invitationCode=18221793699`
- 可见资源列表：`/acx/operation/market/code-resource-info?codeId=18221793699`
- 活动详情：`/acx/operation/market/activity/detail?marketActivityId=v47N2Wvl`
- 活动页前端逻辑：`/ui/acm/assets/index-CE1f7VF5.js`
- 申请页前端逻辑：`/ui/acm/assets/Apply-IlKUkABD.js`
- 资源卡组件：`/ui/acm/assets/ModelCard-CKJRVU_M.js`

说明：活动详情接口的 `relationList` 包含多个区域的资源配置，但该邀请码详情的 `clusterIds` 只有 `11250,11290`，页面实际渲染和申请页实际选择来自 `code-resource-info` 返回的 3 条资源。因此本文的“可选资源”以 `code-resource-info` 为准，计费/额度以这 3 条资源对应的 `relationList` 记录为准。

## 计费方式与试用额度

该活动没有展示正式购买价格，也没有返回价格、单价、金额等字段。页面文案是“邀请您免费试用”“首次申请获取”“为期 90 天体验”，申请页提交类型为 `TRIAL`，所以这里应理解为活动试用额度，而不是按人民币计费。

页面顶部展示的是最高额度：最高 `5000` 核时 / `500` 卡时，为期 `90` 天体验。具体到每个可选资源，申请页会优先使用活动 `relationList` 中与资源匹配的额度：

| 资源 | 算力类型 | 服务类型 | CPU 核时额度 | 异构卡时额度 | GPU_REQUIRE | 存储额度字段 | 试用周期 |
| --- | --- | --- | ---: | ---: | ---: | ---: | ---: |
| `7185-32C-128G` | CPU 通用计算 | 共享型（`SERVICE_TYPE: "SHARED"`） | 5000 | 0 | 0 | 500 | 90 天 |
| `7185-32C-128G-4卡` | 异构计算（4 卡） | 共享型（`SERVICE_TYPE: "SHARED"`） | 2000 | 500 | 0 | 500 | 90 天 |
| `AMD 7542-64C-256G` | CPU 通用计算 | 共享型（`SERVICE_TYPE: "SHARED"`） | 2000 | 0 | 0 | 500 | 90 天 |

字段对应关系来自申请页逻辑：

- CPU 资源提交 `CPU_REQUIRE`，异构额度为 `0`。
- GPU/异构资源提交 `CPU_REQUIRE` 和 `DCU_REQUIRE`，页面中文将异构额度展示为“卡时”。
- 申请页生成的资源信息固定包含 `SERVICE_TYPE: "SHARED"`、`RESOURCE_TYPE`、`RESOURCE_ID` 和 `SPECIFICATIONS`。

活动详情里还存在默认活动字段：

```json
{
  "cpuCup": "10000",
  "cpuDcu": "0",
  "cpuGpu": "0",
  "cpuMemory": "500",
  "gpuCup": "4000",
  "gpuDcu": "1000",
  "gpuGpu": "0",
  "gpuMemory": "500"
}
```

但申请页代码会先按当前选择资源查找 `relationList` 中的 `cpuCore`、`gpuCore`、`storage`，只有找不到匹配记录时才回退到默认值。因此对该链接的实际申请更应采用上表的资源级额度。

## 共享、独占、裸金属判断

### 共享

可以判断为共享资源。申请页提交资源信息时写死：

```json
{
  "SERVICE_TYPE": "SHARED"
}
```

这意味着当前活动申请的是共享服务类型，不是独占节点或裸金属交付。

### 独占

未发现独占资源。活动页、申请页和资源卡组件中未出现 `EXCLUSIVE`、`DEDICATED`、`独占` 等可选服务类型；页面也没有独占计费入口或独占规格切换。

### 裸金属

未发现裸金属资源。虽然资源规格以 CPU、内存、网络、加速卡等节点规格描述，但前端申请参数仍然是共享服务，且未出现 `BARE`、`BAREMETAL`、`裸金属` 等字段或入口。因此不能把这些资源理解为裸金属独占机器。

## 选择建议

- 如果主要跑传统 HPC 科学计算，且希望拿到更高 CPU 核时额度，优先选择华东一区【昆山】`7185-32C-128G`：它有 5000 核时额度和 200Gb IB 网络。
- 如果任务需要 AI、AI4S 或异构加速，选择华东一区【昆山】`7185-32C-128G-4卡`：它是唯一可见的异构/GPU 类资源，额度为 2000 核时 + 500 卡时。
- 如果任务对 CPU 主频更敏感，或需要更大内存单节点规格，可以考虑华北一区【雄衡】`AMD 7542-64C-256G`：主频 2.9GHz、内存 256GB，但试用额度为 2000 核时，网络为 100Gb IB。

## 风险与待确认项

- 该页面没有正式购买价格，不能据此推导转正式购买后的单价。
- `storage: 500` 是接口返回的申请字段，页面未在可见文案中明确单位；可按申请额度字段记录，不建议在未确认前写死为 GB。
- `code-resource-info` 的 `isOpenAutoApprove` 与活动 `relationList` 的 `isAutoApprove` 对个别资源存在不一致，是否自动审批应以后端最终受理结果为准。
