# 中转站设计文档

## 1. 背景

在当前 pre-rollout 接管方案中，VeRL 负责启动和管理 rollout 用的 vLLM 服务，Worker 负责调用模型服务生成 rollout，并计算 episode 结果。多卡运行时，VeRL 可能启动多个 vLLM endpoint，例如：

```text
http://10.10.20.142:40203/v1
http://10.10.20.142:39755/v1
http://10.10.20.142:32829/v1
http://10.10.20.142:36727/v1
```

但 Server/Worker 侧目前更适合接收一个稳定的 OpenAI-compatible model endpoint。为了避免让 Server/Worker 直接理解 VeRL 内部启动了多少个 vLLM 进程，Adapter 侧计划增加一个“中转站”。

## 2. 目标

中转站的核心目标是：对 Server/Worker 暴露一个统一的模型访问地址，对内将请求转发到 VeRL 管理的多个 vLLM endpoint。

主链路如下：

```text
VeRL
  启动多个 vLLM endpoint
    |
    | endpoint 列表
    v
Adapter Model Gateway / 中转站
  对外暴露一个 OpenAI-compatible endpoint
    |
    | EpisodeRequest.payload.model_endpoint.url
    v
Server / Worker
  调用中转站生成 rollout
    |
    | /v1/chat/completions 或 /v1/completions
    v
Adapter Model Gateway / 中转站
  round-robin 转发到多个 vLLM endpoint
```

第一版只做平均分配，也就是 round-robin。后续可以扩展为 least-inflight、健康检查、失败重试、按 token 长度估算负载等策略。

## 3. 职责边界

### 3.1 Adapter 中转站负责

- 启动一个 Worker 可访问的 HTTP 服务，例如 `http://10.10.20.142:18080/v1`。
- 从 VeRL runtime 中获取多个 vLLM endpoint。
- 将 `EpisodeRequest.payload.model_endpoint.url` 填成中转站地址，而不是某个具体 vLLM 地址。
- 接收 Worker 的 OpenAI-compatible 请求。
- 将请求按 round-robin 转发到多个 vLLM endpoint。
- 原样返回 vLLM response。
- 记录请求被转发到哪个 vLLM endpoint、耗时、状态码、失败信息等日志。

### 3.2 Adapter 中转站不负责

- 不负责训练模型。
- 不负责更新模型权重。
- 不负责计算 reward、advantage 或 loss。
- 不负责修改 Server/Worker 的 rollout 逻辑。
- 不负责管理 VeRL 之外的模型服务。

### 3.3 VeRL 负责

- 管理 policy model。
- 在训练 step 之间更新 actor 权重。
- 将最新权重同步到 VeRL 管理的 vLLM rollout 服务。
- 启动一个或多个 vLLM endpoint。

### 3.4 Server/Worker 负责

- Server 负责接收 episode 请求并调度 Worker。
- Worker 负责调用 `model_endpoint.url` 生成 rollout。
- Worker 负责根据 rollout 计算 episode 结果，并返回给上游。

Worker 不需要知道 `model_endpoint.url` 背后是一个真实 vLLM endpoint，还是 Adapter 中转站。

## 4. 接口形态

### 4.1 EpisodeRequest 中传给 Worker 的模型地址

Adapter 构造 EpisodeRequest 时，payload 中的模型地址应指向中转站：

```json
{
  "model_endpoint": {
    "type": "openai_compatible",
    "url": "http://10.10.20.142:18080/v1",
    "model_name": "Qwen/Qwen2.5-0.5B-Instruct",
    "generation_config": {
      "temperature": 1.0,
      "top_p": 1.0,
      "max_tokens": 512
    }
  }
}
```

这里的 `url` 是 Worker 实际访问的地址。Worker 不需要感知后端有几个 vLLM endpoint。

### 4.2 中转站内部维护的 vLLM endpoint 列表

中转站内部维护 VeRL runtime 暴露出的真实 vLLM endpoint，例如：

```json
{
  "upstreams": [
    "http://10.10.20.142:40203/v1",
    "http://10.10.20.142:39755/v1",
    "http://10.10.20.142:32829/v1",
    "http://10.10.20.142:36727/v1"
  ]
}
```

这些地址只属于 Adapter 内部实现细节，不需要传给 Server/Worker。

### 4.3 需要兼容的 OpenAI-compatible 路由

第一版建议支持：

```text
GET  /v1/models
POST /v1/completions
POST /v1/chat/completions
```

请求 body 应尽量原样转发，response 也应尽量原样返回。这样 Worker 侧无需修改现有 OpenAI-compatible client。

## 5. 负载均衡策略

第一版采用 round-robin：

```text
request 0 -> upstream 0
request 1 -> upstream 1
request 2 -> upstream 2
request 3 -> upstream 3
request 4 -> upstream 0
```

这样在 `ROLLOUT_TP=1 + NGPUS_PER_NODE=4` 时，VeRL 通常会启动 4 个单卡 vLLM replica，中转站可以把 Worker 请求平均分配到 4 个 replica，从而让 rollout 阶段更充分地利用 4 张 GPU。

如果使用 `ROLLOUT_TP=4 + NGPUS_PER_NODE=4`，VeRL 可能只启动一个 4 卡 tensor-parallel vLLM endpoint。此时中转站仍然可以工作，但不会发生多 endpoint 分流，相当于透明代理。

## 6. 权重更新一致性

如果多个 vLLM endpoint 是同一个 policy model 的多个 replica，那么每个 replica 都需要在 actor 更新后同步到最新权重。

该同步仍然应该由 VeRL 负责，而不是由 Adapter 中转站负责。中转站只转发请求，不直接加载或更新模型权重。

需要重点关注的是：中转站不能把同一轮 rollout 请求发到权重版本不一致的 endpoint。理想情况下，VeRL 的流程应保证：

```text
actor update 完成
  -> update_weights 同步到所有 vLLM replica
  -> 下一轮 rollout 开始
```

后续可以在日志和请求中补充 `global_step` 或 `policy_version`，用于确认每轮 rollout 使用的模型版本一致。

## 7. 日志与观测

中转站建议记录结构化日志，至少包括：

| 字段 | 说明 |
|---|---|
| `request_id` | 中转站生成或透传的请求 ID |
| `upstream_url` | 本次实际转发到的 vLLM endpoint |
| `route` | `/v1/chat/completions` 等路由 |
| `model` | 请求中的模型名 |
| `status_code` | 上游返回状态码 |
| `latency_ms` | 转发耗时 |
| `stream` | 是否为流式请求 |
| `error` | 失败时的错误信息 |

这些日志可以用来验证多卡分流是否生效。例如 200 个请求在 4 个 upstream 上应接近：

```text
upstream 0: 50
upstream 1: 50
upstream 2: 50
upstream 3: 50
```

## 8. 失败处理

第一版可以采用简单策略：

- 如果某个 upstream 请求失败，记录错误并返回失败。
- 如果 `/v1/models` 请求到某个 upstream 失败，可以尝试下一个 upstream。
- 暂不做复杂重试，避免引入重复生成或长时间阻塞。

后续可以扩展：

- endpoint 健康检查。
- 失败 endpoint 临时摘除。
- 请求级重试。
- least-inflight 调度。
- 超时控制。

## 9. 预期收益

该方案的主要收益是：

- Server/Worker 只看到一个稳定模型地址，接口简单。
- Adapter 屏蔽 VeRL 内部多个 vLLM endpoint 的细节。
- 不需要 Worker 实现多 endpoint 负载均衡。
- 可以在 Adapter 侧统一记录 rollout 模型访问日志。
- 当前 `ROLLOUT_TP=1 + NGPUS_PER_NODE=4` 的多 vLLM replica 能真正被 Worker 使用起来。

## 10. 第一版验收标准

第一版完成后，至少应满足：

- VeRL 启动多个 vLLM endpoint。
- Adapter 中转站启动并监听 Worker 可访问地址。
- EpisodeRequest 中的 `model_endpoint.url` 指向中转站。
- Worker 能通过中转站成功完成 rollout。
- 中转站日志显示请求被平均分配到多个 vLLM endpoint。
- 训练任务可以完成至少 1-step 和 2-steps smoke test。
