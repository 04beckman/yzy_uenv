## 2. VeRL One-step Off-policy 接入

本周将 VeRL 的 one-step off-policy experimental 入口接入到 UEnv pre-rollout 链路。one-step off-policy 的核心是让 rollout 与 update 形成一拍流水线，用轻微 off-policy 换取 step 级并行能力。

当前接入路径如下：

```text
verl.experimental.one_step_off_policy.main_ppo
  -> UEnvAgentLoop
  -> Python Adapter
  -> Rust adapter core
  -> UEnv Server / Worker
  -> EpisodeResult
  -> AgentLoopOutput
  -> VeRL one-step trainer
```

本周完成的主要内容：

| 内容 | 说明 |
|---|---|
| 运行入口 | 新增 `scripts/onestep_offpolicy/run_verl_grpo_onestep_offpolicy_uenv.sh` |
| metadata 注入 | Adapter 根据 `global_step/global_steps` 写入 `parallel_mode=one_step_off_policy`、`global_step` 等字段 |
| 兼容字段 | 保留 `rollout_step`、`policy_version`、`rollout_policy_version`、`parameter_sync_id` 等观测字段 |
| 结果回填 | 保持原有 `response_ids`、`response_mask`、`reward`、`trajectory` 回填逻辑 |

当前设计结论是：one-step 的旧一拍语义由 VeRL trainer 自己保证，Server/Worker 不需要理解或判断 `staleness`，也不应该根据 policy version 主动丢弃样本。

## 3. Adapter 到 Rust Core 的 Streaming 路径

此前稳定链路主要使用同步 batch RPC。为了支持后续更自然的异步训练接入，本周在 Python Adapter 到 Rust adapter core 之间增加了可选 streaming 路径。

| 路径 | 说明 | 当前定位 |
|---|---|---|
| 同步 batch | Adapter 构造 batch 后等待 Rust core 返回完整结果 | 默认稳定路径 |
| streaming | Adapter 通过 stream 发送请求并接收结果 | 异步实验路径 |

这部分改动保持了向后兼容：默认仍走同步 batch，只有在显式启用 streaming 配置时才切换到新路径。这样可以在不影响现有 Layer 4 联调的前提下，继续推进异步能力验证。

## 4. VeRL Fully Async 接入设计

VeRL fully async 的特点是 rollouter 和 trainer 通过 VeRL 内部 `MessageQueue` 解耦。UEnv Adapter 的职责不是重新实现这个 queue，而是让 fully async rollouter 能调用 UEnv，并把结果组织成 VeRL 需要的 `AgentLoopOutput`。

本周完成的主要内容：

| 内容 | 说明 |
|---|---|
| 运行脚本 | 新增 `scripts/fully_async_policy/run_verl_grpo_fully_async_uenv.sh` |
| 最小 request metadata | 当前只要求跨 UEnv 传递 `parallel_mode=fully_async` 和 `global_step` |
| VeRL 内部字段 | Adapter 在 `AgentLoopOutput.extra_fields` 回填 `global_steps/min_global_steps/max_global_steps` |
| logprob 设计 | 明确 `rollout_log_probs` 跨协议类型为 `list[float]`，由 Adapter 转为 VeRL 所需字段 |

当前判断是：fully async 的 stale sample 控制应继续由 VeRL 内部 `MessageQueue` 和 trainer 策略负责。Server/Worker 只负责执行 episode、返回结果、保留 metadata，不参与训练策略判断。

## 5. 异步字段与 Server/Worker 边界

本周重点重写了 `docs/VeRL异步接入UEnv字段设计.md`，将异步字段从“所有可能字段”调整为“新增字段分级”。

当前字段分级如下：

| 分类 | 字段 | 说明 |
|---|---|---|
| 当前必须跨 UEnv 字段 | `parallel_mode`、`global_step` | one-step 和 fully async 的最小字段 |
| 可选观测字段 | `rollout_step`、`policy_version`、`rollout_policy_version`、`parameter_sync_id`、时间戳 | 用于后续排查 policy 版本、权重同步和性能 |
| VeRL 内部字段 | `global_steps/min_global_steps/max_global_steps` | fully async 下回填给 VeRL，不传给 Server/Worker |
| 不应下发字段 | `staleness`、`max_allowed_staleness`、`trainer_policy_version`、`partial_rollout` 等 | 属于训练侧策略或 VeRL 内部语义 |
| 可选结果字段 | `rollout_log_probs` | bypass mode 需要，decoupled mode 可由 VeRL 训练侧重算 |

Server 侧处理要求被收敛为：保留最小 metadata、透传可选 metadata、记录调度时间、支持乱序结果和超时上下文，不做 stale 判断。

Worker 侧处理要求被收敛为：读取并透传 `parallel_mode/global_step`，按能力返回时间字段和 `rollout_log_probs`，不理解也不处理训练策略字段。
