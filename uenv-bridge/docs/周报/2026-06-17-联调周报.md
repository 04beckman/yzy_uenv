## 2. 当前联调链路

当前主链路如下：

```text
VeRL trainer
  -> UEnvAgentLoop / Python shim
  -> Rust adapter core
  -> UEnv Server
  -> UEnv Worker
  -> model endpoint 生成 rollout
  -> reward / trajectory / response
  -> Rust adapter core
  -> Python shim
  -> VeRL AgentLoopOutput
  -> GRPO update
```

本周联调验证的关键点是：VeRL 不再在内部完成 rollout，而是在 pre-rollout 阶段把 prompt-only episode 交给 UENV 链路；Worker 生成 response 并返回 reward、trajectory 等结果后，Adapter 再把结果回填为 VeRL 可训练的 `AgentLoopOutput`。

## 6. 本周定位和解决的问题

|问题|原因|处理结果|
|---|---|---|
|Worker 返回 reward 全为 0|`DATA_MAX_RESPONSE_LENGTH` 过小，response 尚未生成最终答案就被截断|调大 response length 后 reward 恢复正常|
|Worker 是否使用 VeRL 模型|早期对 model endpoint 来源不清晰|明确由 VeRL 管理模型与 vLLM endpoint，Worker 调用 VeRL 暴露的 endpoint|
|Adapter 是否负责模型更新|Worker 侧生成 rollout，但模型权重应由 VeRL 更新和同步|明确 Adapter 不管理模型权重，只传递 endpoint 和回填结果|
|多卡运行时只有一张 GPU 忙|VeRL 启动了多个 vLLM endpoint，但 Adapter 传给 Worker 的 URL 只指向第一个 endpoint|已定位，计划通过中转站解决|
|多卡配置报错|GPU 暴露参数、Ray CUDA 可见性、micro batch size 与并行配置不匹配|已修正运行参数并完成可跑通配置|

## 7. 多卡 rollout 问题与中转站方案

本周在 4 卡配置下发现：VeRL/Ray 确实启动了多个 vLLM server，但 Adapter 当前传给 Worker 的 `model_endpoint.url` 只使用第一个 vLLM endpoint，因此 rollout 请求没有被均匀分摊到多张 GPU。

已形成下一步方案：在 Adapter 侧增加 model gateway / 中转站。

```text
Worker
  -> Adapter 中转站统一 endpoint
  -> round-robin 转发到多个 VeRL vLLM endpoint
```

该方案的优势是：

- Server/Worker 仍然只看到一个 OpenAI-compatible endpoint。
- 多个 vLLM endpoint 的发现和分流由 Adapter 屏蔽。
- 不需要 Worker 实现多 endpoint 负载均衡。
- 后续可以在中转站统一记录 rollout 请求日志、耗时和 upstream 分布。

详细设计见 `中转站设计文档.md`。

## 8. 当前结论

本周已经证明 UENV Adapter 主链路不仅完成接口联通，而且可以支撑真实 Worker 参与的 GRPO 训练。当前链路能够完成 1-step、2-steps、20-step 和双并发 2x20 联调，Adapter 能稳定处理 response、reward、trajectory 和 VeRL 训练回填。
