## 2. 分布式 Layer 4 运行脚本

本周继续完善真实 VeRL + Server/Worker 的 Layer 4 联调脚本。当前主脚本为：

```text
scripts/run_layer4_distributed.sh
```

该脚本当前承担以下职责：

- 启动 VeRL 训练容器。
- 配置 UENV AgentLoop，在 pre-rollout 阶段接管 VeRL episode。
- 连接 server-side adapter core。
- 配置训练步数、batch size、rollout 参数、PPO micro batch 等训练参数。
- 配置 Podman 暴露 GPU 与容器内 `CUDA_VISIBLE_DEVICES`。
- 将运行日志、request/result 记录到统一目录。

本周重点修正了多 GPU 运行中容易出错的参数关系。例如宿主机 GPU 暴露和容器内编号需要分开理解：

| 参数 | 作用 |
|---|---|
| `PODMAN_GPU_ARGS` | 控制宿主机哪些 GPU 被暴露给容器 |
| `CUDA_VISIBLE_DEVICES_IN_CONTAINER` | 控制容器内 VeRL/Ray 看到的 GPU 编号 |
| `NGPUS_PER_NODE` | 告诉 VeRL 当前训练任务可使用多少张 GPU |
| `PPO_MINI_BATCH_SIZE` | GRPO/PPO 更新时的 mini batch 大小 |
| `PPO_MICRO_BATCH_SIZE_PER_GPU` | 每张 GPU 上的 micro batch 大小 |

处理后，脚本可以更稳定地在单卡、双卡和多卡配置间切换。

## 3. Adapter Model Gateway / 中转站

在多 GPU 训练中，VeRL 可能启动多个 vLLM endpoint。如果 Adapter 只把第一个 endpoint 传给 Worker，那么 Worker 的 rollout 请求实际只会打到一个 vLLM 上，多 GPU rollout 吞吐无法充分利用。

为解决这个问题，本周在 Adapter 侧增加了 model gateway，也就是中转站逻辑：

```text
Worker
  -> Adapter model gateway
  -> round-robin 转发到多个 VeRL vLLM endpoint
```

当前中转站的第一版目标比较明确：

| 目标 | 说明 |
|---|---|
| 统一模型入口 | Worker 只需要访问一个 OpenAI-compatible endpoint |
| 屏蔽多个 vLLM endpoint | Adapter 内部维护 upstream 列表 |
| 简单负载均衡 | 第一版采用 round-robin 平均分发 |
| 保留观测能力 | 后续可以记录每个 upstream 的请求量、延迟和失败情况 |

相关实现主要包括：

| 文件 | 内容 |
|---|---|
| `src/uenv/bridge/model_gateway.py` | OpenAI-compatible 请求转发与 upstream 轮询 |
| `scripts/verify_model_gateway_loop.py` | 本地回路验证脚本 |
| `scripts/run_layer4_distributed.sh` | 接入 gateway 启动和配置逻辑 |
| `configs/uenv-agent-loop.yaml` | 增加 gateway 配置项 |

当前这部分已经具备本地回路验证能力。后续需要结合真实 Server/Worker 请求，进一步验证多个 vLLM endpoint 是否真正被均匀使用。

## 4. Batch 级接入

当前稳定链路仍然可以逐条 episode 发送到 adapter core；但当训练规模较大时，逐条发送会带来明显的调度开销。本周在 Adapter 侧补充了 batch 级接入能力。

当前思路是在 VeRL 更高层的 rollout manager / AgentLoopWorker 位置拿到整批样本，再由 Adapter 构造多个 episode request，按 batch 发送给下游。

主要涉及文件如下：

| 文件 | 作用 |
|---|---|
| `src/sitecustomize.py` | 根据环境变量决定是否启用 VeRL batch patch |
| `src/uenv/bridge/verl_batch_agent_loop_patch.py` | 在 VeRL rollout manager / AgentLoopWorker 层获取整批样本 |
| `src/uenv/bridge/verl_agent_loop.py` | 增加 batch 请求构造、chunk 切分和重试逻辑 |
| `configs/uenv-agent-loop.yaml` | 增加 batch size、重试次数等配置项 |

## 6. Step 级并行初步调研

由于当前主要瓶颈在 rollout 阶段，本周开始关注 VeRL 是否提供 step 级并行能力。初步调研发现，VeRL experimental 中至少有两类相关策略：

| 策略 | 核心思想 |
|---|---|
| one-step off-policy | 下一批 rollout 可以和当前批训练更新重叠执行，用轻微 off-policy 换取流水线并行 |
| fully async | rollout 和 trainer 通过异步队列解耦，trainer 持续消费 rollout 样本并进行更新 |

这两类策略的共同目标都是减少同步 step 中“先 rollout、再 update”的串行等待时间。但它们对资源切分、样本新鲜度、参数同步和训练稳定性都有额外要求，因此不能只从理论上判断是否更快，需要后续用真实脚本对比。

下周计划之一是补充对应运行脚本，在相近配置下真实测试同步训练、one-step off-policy 和 fully async 的执行时间差异。

- VeRL pre-rollout AgentLoop 接管。

- 支持记录 request/result 和训练日志。
- 支持训练日志解析与图表化分析。
- 具备 batch 级接入的 Adapter 侧代码基础。
