## 2. 文本阅读理解：PubMedQA

本周完成 PubMedQA expert-labeled 数据集的基线评测。数据集共 1000 条样本，标签为 `yes/no/maybe`，主指标为 Accuracy 和 Macro-F1。

当前结果如下：

| 口径 | 样本数 | Parse rate | Accuracy | Macro-F1 | 说明 |
|---|---:|---:|---:|---:|---|
| vLLM generate | 1000 | 1.0000 | 0.7980 | 0.6202 | 直接调用目标模型，自由生成后解析标签 |
| vLLM label_logprob | 1000 | 1.0000 | 0.6780 | 0.4749 | 候选标签打分，`maybe` 类召回较差 |
| UEnv no-thinking 全量 | 1000 | 0.9990 | 0.7960 | 0.5718 | AdapterCore / Server / Worker / gateway 全链路 |
| UEnv thinking 子集 | 10 | 1.0000 | 0.7000 | 0.4786 | 开启 thinking，`MAX_TOKENS=512` |

这部分工作证明 PubMedQA 可以复用 `env_type=math`，通过 `env_config.dataset=pubmedqa` 路由到 Worker 侧 PubMedQA backend。UEnv no-thinking 全量结果与直接 vLLM generate 结果接近，说明 Adapter 组包、Server 调度、Worker 判分和 result 聚合链路基本正确。

## 4. 数学题求解：OlymMATH

本周完成 OlymMATH EN/ZH EASY/HARD 四个公开文件的基线评测。数据集共 400 条题目，主指标包括 Accuracy、Parse rate、Parsed accuracy、分语言/难度指标，以及官方对齐口径下的 Pass@10。

当前核心结果如下：

| 口径 | 题目数 | Generation samples | Parse rate | Accuracy / Sample accuracy | Problem parse rate | Pass@10 |
|---|---:|---:|---:|---:|---:|---:|
| single-sample baseline | 400 | 400 | 0.8175 | 0.5025 | - | - |
| 官方对齐 thinking 口径 | 400 | 4000 | 0.7468 | 0.5128 | 0.9625 | 0.7775 |
| UEnv boxed no-thinking 全量 | 400 | 400 | 0.8750 | 0.1325 | - | - |

其中官方对齐口径开启 thinking，`MAX_TOKENS=32768`，每题采样 10 次。该口径下 problem-level Pass@10 达到 77.75%，说明模型经常能在多次采样中至少答对一次，但 sample-level 稳定性仍不足。

UEnv 口径方面，本周补充了两类现象：

| UEnv 口径 | 样本 | 结果 | 结论 |
|---|---:|---|---|
| `boxed_no_think` + `MAX_TOKENS=256` | 400 | completed 400/400，Accuracy 0.1325 | 链路稳定，但短答案约束不代表正式数学能力 |
| `official` + thinking + `MAX_TOKENS=512` | 1 | completed 1/1，Parse rate 0 | 能返回 EpisodeResult，但 512 token 内没有最终答案 |
| `official` + thinking + `MAX_TOKENS=2048` | 1 request | gateway 多次收到 vLLM 生成请求，但本地 client 未拿到 EpisodeResult | 长 thinking 下需要继续定位 Worker/Server 的判分、重试或返回逻辑 |

## 8. 接入 UEnv 的运行流程

本周评测中同时保留了两条口径：一条是直接调用 vLLM 的基线评测，另一条是接入 UEnv 后的环境评测。接入 UEnv 后，不再由本地评测脚本直接调用模型并自行判分，而是将样本包装成 `EpisodeRequest`，交给 AdapterCore / Server / Worker 完成模型调用、环境执行和 reward 计算。

整体流程如下：

```text
Benchmark 数据集
  -> benchmark driver 读取样本
  -> Adapter 构造 EpisodeRequest
       - env_type: math / code / swe
       - env_config.dataset: pubmedqa / scitab / olymmath-easy / ...
       - model_endpoint.url: adapter model gateway 或真实模型 endpoint
       - reward_config.target: 标准答案 / 标签 / 测试目标
  -> Rust AdapterCore
  -> UEnv Server
  -> Worker 按 env_type 和 dataset 路由到对应 backend
  -> Worker 调用模型 endpoint 生成 response
  -> Worker 执行解析、判分或测试
  -> Worker 返回 EpisodeResult
  -> Adapter 聚合 response / reward / trajectory
  -> benchmark driver 统计 Accuracy、Macro-F1、pass@1、Pass@10 等指标
```

其中 Adapter 侧主要负责三件事：

| 阶段 | Adapter 侧工作 | 说明 |
|---|---|---|
| 组包 | 将 benchmark 样本转换为 `EpisodeRequest` | 补齐 `env_type`、`env_config.dataset`、`reward_config.target`、`model_endpoint` 等字段 |
| 发送 | 通过 Rust AdapterCore 调用 UEnv Server | 当前 PubMedQA / SciTab / OlymMATH 已走该路径 |
| 汇总 | 将 `EpisodeResult` 转回评测记录 | 从 result 中读取 response、reward、status、trajectory 信息，并统计任务指标 |

Server / Worker 侧主要负责：

| 模块 | 工作 |
|---|---|
| Rust AdapterCore | 接收 Adapter 请求，转换为 Server 可处理的 episode 请求，并等待结果返回 |
| UEnv Server | 根据 `env_type` 调度到对应 Worker / plugin |
| Worker | 调用模型 endpoint，执行 benchmark-specific parser / reward / tester，并返回 `EpisodeResult` |
| Model gateway | 将 Worker 的 OpenAI-compatible 请求转发到本机 vLLM，同时可控制 thinking 开关和记录模型调用日志 |

以 PubMedQA 为例，本周 UEnv 全量评测的单条样本流程为：

```text
PubMedQA question + context + gold label
  -> env_type=math, dataset=pubmedqa, target=yes/no/maybe
  -> AdapterCore / Server
  -> math Worker 的 PubMedQA backend
  -> Worker 通过 gateway 调用 Qwen/Qwen3.6-35B-A3B
  -> backend 解析 yes/no/maybe 并计算 reward
  -> EpisodeResult 返回 Adapter
  -> driver 汇总 Accuracy / Macro-F1 / reward accuracy
```

这条流程与直接 vLLM 评测的区别在于：直接 vLLM 评测只验证“模型在数据集上的原始能力”，而 UEnv 评测额外验证了 Adapter 组包、Server 调度、Worker 环境执行、reward 计算和 result 返回是否都能闭环。因此 UEnv 口径的结果不仅是模型分数，也是系统联调结果。
