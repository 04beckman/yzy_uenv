## 2. 五类 Benchmark UEnv 基线

本周将五类任务统一到 UEnv 全链路评测口径：Adapter driver 读取数据集并构造 `EpisodeRequest`，Server/Worker 完成模型调用、环境执行和 reward 计算，Adapter 回收 `EpisodeResult` 并统计指标。

| Benchmark | 数据规模 | UEnv 状态 | 主指标 | 当前结果 |
|---|---:|---|---|---:|
| PubMedQA | 1000 | completed 1000 / failed 0 | Accuracy / Macro-F1 | 0.8000 / 0.5912 |
| SciTab | 1224 | completed 1224 / failed 0 | Accuracy / Macro-F1 | 0.7451 / 0.7340 |
| DSCodeBench | 1000 | completed 1000 / failed 0 | pass@1 | 0.2670 |
| OlymMATH | 400 | completed 400 / failed 0 | reward accuracy / parse rate | 0.6575 / 0.9500 |
| SWE-bench-Pro | 731 | 当前返回 184 条 | resolved | 暂无有效最终分数 |

结论：分类、表格、数学和代码生成任务已形成可复现的 UEnv baseline；SWE-bench-Pro 链路可以返回 result，但当前 `resolved=true` 仍为 0，需要继续定位 OpenHands 工作目录、patch 收集和官方测试执行一致性。

## 3. 结果与问题

PubMedQA 和 SciTab 的 Adapter 解析结果与 Worker reward 一致，说明 request 组包、gateway 模型调用、Worker 判分和 result 聚合链路稳定。PubMedQA 的主要短板是 `maybe` 类召回偏低；SciTab 中 `not enough info` 召回偏低，模型仍倾向给出确定性判断。

OlymMATH 使用 `MAX_TOKENS=32768`、`THINKING_TOKEN_BUDGET=16384`、`strip_reasoning=true` 的正式 thinking 口径。初始全量中 22 条 EN-EASY 样本 failed，随后通过 `RESUME=1` 只重测失败样本，最终 400 条全部 completed。结论：全量请求、失败补测和结果聚合链路闭合，但仍建议 Worker 后续回填 `finish_reason`、retry attempt 和原始错误，便于区分模型输出问题和服务稳定性问题。

DSCodeBench 完成 1000 条全量 code env 评测，`pass@1=0.2670`。当前 Worker 的 `inline_harness` 在失败样本上多以 `AssertionError` 形式返回，因此本轮 `execution_rate` 更接近“成功执行且通过比例”，后续需要 Worker 返回结构化的 `tests_run`、`tests_passed` 和失败原因，才能进一步拆分运行失败、答案错误和环境异常。

SWE-bench-Pro 当前 UEnv 结果文件已有 184 条返回，其中 completed 146、failed 38、`resolved=true` 为 0。已归档的诊断重点包括：部分样例可能创建或修改了与目标 repo 不一致的路径；OpenHands 轨迹、最终 `git diff`、测试执行目录和 official resolved 结果仍需 Worker/Agent 侧核验。

## 4. 运行口径统一

本周将五类 benchmark 文档中的运行方式统一为三步：

```text
启动 8GPU vLLM
  -> 启动 Adapter Model Gateway
  -> 运行对应 run_*_uenv_baseline.sh
```

其中 PubMedQA、SciTab、DSCodeBench、OlymMATH 直接通过 `UENV_ROLLOUT_MODEL_ENDPOINT` 把 gateway URL 传给 Worker；SWE-bench-Pro 额外依赖 OpenHands 的 `LLM_CONFIG_PATH`，该 JSON 中的 `base_url` 也必须指向同一个 Adapter Model Gateway。
