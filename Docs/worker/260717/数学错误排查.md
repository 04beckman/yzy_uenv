## 排查结论

主故障在 **Worker 的 Math 插件 `step()` / OlymMATH 评分阶段**，不是 Model Gateway，也不是 Server→Worker 网络中断。

完整链路已经对齐：

1. Server 成功把 Episode 派发给 `worker-7143-pro`。
2. Worker 成功获取 warm Math 插件实例。
3. Worker 调用模型，日志明确记录 `model_http_ok` 和 `model_response_ok`。
4. 模型返回后约 1 ms，Worker 调用 Math 插件 `step()` 失败：
   ```text
   phase=step
   error_code=ERR_ENV_STEP_FAILED
   h2 Reset(StreamId(1), CANCEL, Remote)
   ```
5. Worker 将其包装成 `execute_episode_failed`。
6. Server 将它包装成 `dispatch_failed` 并重试。
7. 三次均确定性失败，最后才返回错误码 `5001 / exceeded max attempts (3)`。

最有力的对账是：

```text
251 个成功样本 × 1 次模型请求
+ 149 个失败样本 × 3 次模型请求
= 698 次模型请求
```

这与 Gateway 的 `698 次 HTTP 200` 完全一致。说明所有失败样本的三次模型调用实际上都成功了，失败全部发生在模型返回之后。

## 具体证据

EN 失败样本 `...EASY-51-EN...`：

- 模型耗时：约 `79.6s`
- 三次返回内容大小均为 `2274 bytes`
- 每次 `model_response_ok` 后立即在 `step()` 出现 h2 CANCEL

ZH 失败样本 `...EASY-0-ZH...`：

- 模型耗时：约 `129.3s`
- 三次返回内容大小均为 `14933 bytes`
- 同样每次在模型成功后立即 `step()` 失败

TSV 也显示明显的内容相关性：

- 前 200 个样本仅失败 2 个
- 200–399 失败 147 个
- 索引 200 正好进入中文数据
- 失败中 ZH 占 `147/149`

因此它不是随机网络故障，更像 Math 插件评分代码对部分中文、LaTeX、长输出或特殊答案格式触发了 panic/处理任务异常。tonic 没有收到正常的应用层 `Status`，最终表现为远端 `RST_STREAM CANCEL`。

`pool.cool_timeout=60` 已排除：active lease 不会被该逻辑回收，而且相邻成功样本模型耗时也超过 80 秒。

## 责任划分

- **主要缺陷：Worker/Math 插件**
  - `uenv-math-plugin` 的 `step()` 中调用 `score_action(...)`
  - 评分异常没有被转换成结构化错误，而是导致 h2 流被取消
  - 插件 stderr 当前被重定向到 `/dev/null`，panic 原因没有留下日志

- **次要缺陷：Server 重试策略**
  - Server 把确定性的 `ERR_ENV_STEP_FAILED` 当成可重试的 `dispatch_failed`
  - 每个失败题额外重复两次昂贵的模型生成
  - 149 个失败因此额外产生 `298` 次模型请求

- **无关噪声**
  - Worker 日志中还有旧 WAL 的 `DUPLICATE_REJECTED / pending result not found`
  - 它与本次 `step()` h2 CANCEL 不在同一失败链，不是这 149 个错误的直接原因

建议下一步修复 `score_action`：排除 UTF-8 字节切片、`unwrap/assert`、不安全 LaTeX 提取等 panic 点；同时让插件捕获评分异常并返回明确的 gRPC `Status`。Server 应把确定性评分错误标记为不可重试，避免再次调用模型。
