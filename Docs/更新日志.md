## 文档编码注意事项（PowerShell / Windows）

2026-05-25 起，`Docs/` 下 Markdown 曾因 **PowerShell 写入未指定编码** 出现 UTF-8 损坏（如 [`worker-pool-mvp-checklist.md`](./worker-pool-mvp-checklist.md) 乱码、本文件部分条目乱码）。后续修改请遵守：

1. **文件编码**：`Docs/` 中文 Markdown 统一 **UTF-8（无 BOM）**。
2. **读通常安全**：`Get-Content` 只读一般不会破坏文件。
3. **写必须显式编码**：`Set-Content`、`Add-Content`、`Out-File` 等**重写字节**的命令必须带 `-Encoding utf8NoBOM`（PowerShell 7+）或等效 UTF-8 无 BOM 策略；**禁止**依赖系统默认编码。
4. **已确认高风险命令**（未带 `-Encoding` 时）：
   - `Set-Content -Path 'Docs\worker-pool-mvp-checklist.md' -Value ($head + $mid + $tail)`
   - `... | Add-Content -Path 'Docs\更新日志.md'`
5. **损坏形态与处置**：
   - 第一阶段：UTF-8 多字节字符**最后一字节变为 `?`**；
   - 第二阶段：若用脚本“猜续字节”修复，会产生**形似中文的错字**（如「骨枀」「插什」）——**禁止**再用此类脚本；
   - 正确做法：以未损坏副本（如 `worker-pool-mvp-checklist_1.md`）或版本历史合并还原。
6. **修改后自检**：保存后打开文件或 `grep` 抽查中文关键词，确认无 `?`、无错字、Markdown 符号（`§`、`→`、`**`）正常。
7. **本次事故（2026-05-25，本文件全文 `?`）**：
   - **现象**：对 [`更新日志.md`](./更新日志.md) 做内容整理（时间倒序重排）后，**全部中文变为 `?`**，英文、日期、路径、代码标识仍可读。
   - **推断根因**：文件被**以非 UTF-8 编码（多为 Windows 系统默认 ANSI/GBK）或错误解码后再写入**整文件覆盖；UTF-8 多字节序列无法映射到目标单字节页时，被替换为 `?`（典型“不可逆全损”，不同于仅末字节损坏）。
   - **高风险场景**：自动化工具 / Agent **批量重写** `.md` 时未声明 UTF-8；PowerShell `Set-Content` / `Out-File` 无 `-Encoding`；将已为 `?` 的文件当作“已修复”再次保存；**对中文文件名的目标路径直接写入时也可能触发错误编码管道**（本文件两次复现）。
   - **处置**：**禁止**在 `?` 占位文本上继续编辑或“猜字修复”；须用未损坏副本、Git 历史、或 `python scripts/restore_update_log.py` 等**显式 UTF-8 写出**整文件恢复。
   - **预防**：保存前目视标题与首段中文；大段改写后立刻 `grep` 关键词（如 `更新日志`、`M1.4`）；优先用编辑器“UTF-8”保存；脚本写入用 `path.write_text(..., encoding='utf-8')` 或 PowerShell `-Encoding utf8NoBOM`。

## 更新规则

1. **按时间倒序**：最新日期在最上方；同一日期内按时间倒序（新的在上）。
2. **日期 + 时间**：每条记录使用二级标题 `## YYYY-MM-DD HH:MM`（本地时间，24 小时制）。
3. **按条列举**：每条变更一行，使用 `-` 列表；一条一事，写清「做了什么 / 影响范围」。
4. **只记已完成**：进行中的工作不写；合并 PR 或本地验收通过后再追加。
5. **可引用文档**：若变更依据某设计/清单，在条目中用相对链接注明（如 `worker-pool-pre-mvp-architecture-adjustment.md`）。
6. **不写 MVP 业务细节占位**：框架级骨架与契约对齐记本日志；Mock 主动 Dispatch、Episode 执行等运行时逻辑记对应里程碑完成时再追加。
7. **追加条目时的编码**：向本文件追加内容时，**禁止**使用未指定 `-Encoding` 的 `Add-Content`；优先用编辑器保存，或 `Add-Content -Encoding utf8NoBOM`；或由 `restore_update_log.py` 合并后整文件写出。
