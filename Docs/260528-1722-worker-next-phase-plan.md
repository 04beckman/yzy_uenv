# Worker 层下一阶段补齐规划（基于 design 对照）

> **文档版本**：v2.0  
> **更新时间**：2026-05-28 17:30  
> **定位**：在“Worker 层 MVP 已完成”的前提下，按 [worker-pool-layer-design.md](./worker-pool-layer-design.md) 与 v7.1 相关目标，明确下一阶段功能缺口与补齐计划。  
> **适用范围**：仅覆盖 Worker 层与其直接联调边界（Server/Hub 的跨层依赖单列说明）。

---

## 1. 现状判定（本规划起点）

- 当前可确认：Worker 层 MVP 闭环已具备（控制面链路、预热池、基础容错与可观测、预联调回归）。
- 当前未闭环：真实跨机 Server 联调验收、Hub 真实接入、多环境与沙箱化执行能力。
- 规划目标：从“单环境 MVP 可用”升级到“多环境可负载 + 跨层可验收 + 通用环境可扩展”。

## 3. 下一阶段功能缺口与补齐计划

## 3.1 P0：真实链路验收收口（M7 收尾）

### 缺口

- 目前仅完成本机预联调，缺真实跨机 `uenv-server` 验收闭环。
- 缺少双侧日志交叉证据（Register/Heartbeat/Dispatch/Report）。

### 计划

- 在真实网络路径完成至少 1 轮 GSM8K 与 1 轮新增 MathEnv 请求联调。
- 固化联调记录模板（请求 ID、Worker endpoint、Server 日志定位键、Worker 日志定位键）。

### 验收标准

- 关闭 checklist 中 M7 最后一项待办。
- 输出“跨机联调证据记录”文档（含时间、环境、结果、异常与处置）。

---

## 3.2 P0：MathEnv 与通用数学环境能力

### 设计依据

- design §3（插件化多语言实例）、§9.1（Phase 0 环境收敛来源于 MathEnv 路径）、§4.2（Episode 主循环）。

### 当前缺口

- 当前稳定能力仍以 `gsm8k` 为主，缺少通用 `env_type=math` 的能力分层与可扩展输入输出约定。
- 缺“数学计算类环境”统一承载模型（不同数据集/题型/求解器仍未抽象到统一环境能力）。

### 计划（按功能，不写底层细节）

1. 定义 `math` 环境能力边界：  
   - 统一请求字段（题目、上下文、约束、目标类型）  
   - 统一结果字段（答案、解释、判分依据、状态码）
2. 将 `math` 纳入 Worker 支持环境类型并接入预热池：  
   - Worker 启动可初始化 `math` 预热实例  
   - Dispatch 按 `env_type` 路由到 `math` 实例池执行
3. 支持“数学计算类环境实例”负载：  
   - 在同一 Worker 中可并行承载 `gsm8k` 与 `math`  
   - 统计分环境命中率、失败率与耗时

### 验收标准

- `math` 请求可从 Dispatch 到 Report 完整跑通。
- `gsm8k`/`math` 混合负载下无实例串扰、无双分配。
- 文档中给出 `math` 配置样例与最小回归矩阵。

## 3.3 P1：沙箱与通用环境执行能力（Code/Agent 场景前置）

### 设计依据

- v7.1 对 CodeEnv/AgentEnv 的扩展方向；design §11.1（step 重试边界）、§4.4（资源隔离）、§6（后端演进）。

### 当前缺口

- 目前尚无可运营的“沙箱执行能力”规范化接入（尤其是有副作用工具调用场景）。
- 缺少通用环境分级：只读工具、受限写工具、高风险外部调用的执行策略分层。

### 计划

1. 定义 Worker 可识别的环境执行级别：  
   - `read_only`（可安全重试的只读调用）  
   - `side_effecting`（有副作用调用，禁止自动 step 重试）  
   - `isolated`（必须在受限沙箱或容器边界执行）
2. 明确通用环境接入准入条件：  
   - 能力声明（所需工具、网络、文件系统权限）  
   - 失败语义（可重试/不可重试/必须人工介入）
3. 逐步引入沙箱承载路径：  
   - 先在测试环境验证隔离级别策略  
   - 再与后端（含 Podman 路径）联合验收

### 验收标准

- 有副作用 step 在默认策略下不被自动重试（保持 design §11.1）。
- 通用环境请求可按执行级别进入对应策略路径并可观测。

---

## 3.4 P1：Hub 协同接入（环境发现与能力同步）

### 设计依据

- design §3.6（Hub 元数据协同、manifest 生命周期）。

### 当前缺口

- 目前依赖本地静态 manifest，缺少来自 Hub 的统一能力来源与版本协同。

### 计划

1. 与 Hub 层对齐最小字段：`env_type`、版本、能力标签、后端要求、资源画像。  
2. 约定 Worker 侧同步行为：启动拉取/增量更新/失败回退。  
3. 保持兼容：Hub 不可用时退回本地 manifest，不阻断 Worker 启动。

### 验收标准

- Worker 能识别并加载来自 Hub 的环境能力信息。
- Hub 不可用时可自动降级回本地配置。

---

## 3.5 P2：预热池动态化与容量治理

### 设计依据

- design §5.4（`WarmupSizer`）、§10.2（指标基础）。

### 当前缺口

- 仍使用固定参数，无法根据负载动态收敛。

### 计划

- 实现动态容量策略：基于命中率、请求速率、活跃执行量调节目标池大小。
- 增加参数护栏：扩缩容步长、冷却窗口、最大最小池界限。

### 验收标准

- 在波动负载下，命中率与启动延迟相较固定池策略可量化改善。

---

## 4. 推荐实施顺序（下一阶段）

1. **先收口联调**：真实跨机 M7 验收闭环。  
2. **再扩环境**：MathEnv + 多环境负载执行。  
3. **再补通用能力**：沙箱分级与通用环境准入。  
4. **同步跨层**：Hub 最小协同接入。  
5. **最后优化**：动态预热与生产化指标增强。

---

## 5. 下一阶段完成定义（DoD）

- Worker 可稳定负载 `gsm8k + math` 两类环境请求。
- 完成真实 `uenv-server` 跨机链路验收并留痕。
- 形成“通用环境执行级别 + 沙箱策略”文档并落到回归测试。
- Hub 协同最小闭环可用（可降级）。
