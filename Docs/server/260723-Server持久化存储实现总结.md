# UEnv Server 持久化存储实现总结

日期：2026-07-23  
分支：`bridge-alignment`  
实现基线：`b4798f9db9fb49fb0c95fa483a60f02ef7f49655`  
当前状态：代码已实现并完成隔离验证，尚未提交、安装或部署。

## 1. 背景与目标

原 UEnv Server 的任务状态、Worker 派发关系、幂等记录和 AgentJob 状态主要保存在进程内存中。Server 重启后，排队任务、执行中任务和已完成结果的内存索引会丢失，客户端只能重新提交，存在任务丢失、重复执行和迟到结果无法确认等风险。

本次改造的目标是把 Server 从纯内存调度器改造成可恢复的持久化状态机，并保持现有 gRPC 接口和 Worker 执行路径不变。核心要求包括：

1. Server 重启后恢复排队、执行中和已完成任务。
2. Worker 已完成但尚未得到 Server 确认的结果不能丢失。
3. 相同任务重复提交时不能重复执行。
4. 错误 Worker、旧 attempt、旧 lease 或伪造 token 的结果不能污染当前任务。
5. 持久化默认开启，数据库路径和容量限制可配置。
6. 改造和验证不能影响当前线上 `/usr/local/bin/uenv-adapter-core`。

## 2. 已实现的主要功能

### 2.1 Server 运行状态持久化

新增 SQLite 状态库，默认配置路径为：

```text
/home/uenv/server-state/server-state.db
```

数据库记录任务从提交到终态的完整状态，包括：

- 原始 `EpisodeRequest` 和请求校验和；
- submission kind、batch、parallel mode 和 deadline；
- `queued`、`dispatch_intent`、`dispatched`、`completed`、`failed`、`cancelled`、`timed_out` 等阶段；
- attempt、Worker、dispatch lease、dispatch token 和 Server epoch；
- 最终 `EpisodeResult`、结果校验和、错误码和错误信息；
- ReportResult 幂等记录和迟到结果处理结果；
- AgentJob、Gateway session 和 outbox 状态。

### 2.2 持久化顺序和事务边界

关键外部动作采用“先持久化、后执行”的顺序：

```text
任务提交
  → 先写 episodes(queued)
  → 再进入调度

任务派发
  → 先写 dispatch_intent
  → 再调用 Worker
  → Worker 接受后写 dispatched

任务完成
  → EpisodeResult、幂等记录、终态结果、dispatch 完成状态和 outbox
    在同一个 SQLite 事务中提交
  → 再向等待方返回结果
```

进程在事务中间崩溃时，SQLite 保证事务整体提交或整体回滚，不会只写入一半终态。

### 2.3 Episode 单飞与重复提交处理

新增 `EpisodeCoordinator`，以 `episode_id + request_checksum` 管理同一任务的并发提交：

- 第一个请求成为 owner，负责启动真实执行；
- 后续相同请求附着到同一个 coordinator，等待同一终态；
- 相同 `episode_id` 携带不同请求内容时返回 `EPISODE_ID_CONFLICT`；
- 数据库中已经存在终态时，直接返回持久化结果，不再调用 Worker。

### 2.4 派发身份和迟到结果校验

每次派发记录以下身份信息：

- `episode_id`；
- `attempt_id`；
- `dispatch_lease_id`；
- `dispatch_token`；
- `worker_id`；
- `server_epoch`。

Worker 上报结果时必须带回匹配信息。Server 重启后 epoch 会变化，但旧 Worker 结果只有在数据库中能够找到完全匹配的旧 dispatch、Worker 和 token 时才会被接受。

对于已完成、已取消、已超时或重复上报的结果，Server 返回稳定的幂等响应，例如：

- `DUPLICATE_ACCEPTED`；
- `ALREADY_COMPLETED`；
- `LATE_AFTER_CANCEL`；
- `LATE_AFTER_TIMEOUT`；
- `IDEMPOTENCY_CONFLICT`。

### 2.5 Worker 结果 WAL

Worker 在任务执行完成后，先把待上报结果写入本地 WAL，再向 Server 调用 `ReportResult`：

```text
Worker完成任务
  → 写入 *.wal
  → 向Server上报
  → Server持久化并返回ACK
  → Worker删除对应WAL
```

如果 Server 暂时不可用，WAL 保留。连接恢复后，Worker 使用原来的幂等键、dispatch lease 和 token 重放结果。这样可以覆盖“Worker 已完成、Server 在确认前重启”的窗口。

Worker 注册时同时上报当前 load 和 max load，使重启后的 Server 可以恢复 Worker 容量视图。

### 2.6 Server 启动恢复

Server 启动时读取所有非终态记录并按阶段分类处理：

- `queued`：重新进入调度流程；
- 已存在 dispatch：恢复原 attempt、lease、token、Worker 和 deadline，重建等待关系，接收 Worker WAL 重放；
- 已完成任务：从 SQLite 直接返回终态结果；
- AgentJob：恢复 pending/leased 状态及 coordinator；
- Gateway session：继续执行未完成的清理；
- outbox：重放未投递结果并标记已投递。

恢复逻辑不会把已经有 dispatch 的任务直接当成新任务执行，从而降低重复执行风险。

### 2.7 非确定性失败退避

恢复测试发现，Worker 注册与派发监听端口完全就绪之间存在很短的时间窗口。原代码对 transport error 立即循环重试，可能在数毫秒内耗尽全部 attempt。

现已在非确定性派发失败后使用 `schedule_retry_interval_ms` 做可取消退避。确定性的环境执行失败仍然直接形成终态，不会重复执行昂贵任务。

### 2.8 稳定的 protobuf 校验和

恢复测试还发现，prost 对 protobuf `map` 字段使用 `HashMap` 迭代顺序编码。相同逻辑结果在 clone、WAL 解码或进程重启后，字节顺序可能不同，原始字节哈希会把相同结果误判为冲突。

现已实现稳定校验和：

- 普通 protobuf 字段按原格式编码；
- `EpisodeRequest.metadata` 按 key 排序后加入校验和；
- `EpisodeResult.metadata` 和每个 `StepRecord.info` 按 key 排序后加入校验和；
- 重复终态提交同时使用 protobuf 语义相等判断。

### 2.9 容量保护、健康状态和自动清理

持久化配置增加以下保护：

- 数据库文件最大值；
- SQLite WAL 最大值；
- 最小磁盘剩余空间；
- 单条结果最大值；
- 写入队列容量；
- 终态和幂等记录 TTL；
- 已完成任务最大保留数量；
- shutdown grace。

后台每 30 秒执行：

- 清理过期幂等记录；
- 清理过期 terminal outcome；
- 清理已投递且过期的 outbox；
- 清理超过 TTL 的终态任务；
- 按 `max_completed_entries` 限制已完成任务数量；
- 重试 Gateway 清理；
- 重放 pending outbox。

管理状态中增加持久化健康信息。数据库、WAL 或剩余空间越界时，写入器进入 unhealthy 状态，避免在无法可靠落盘时继续静默接收任务。

## 3. 数据库表

| 表 | 作用 |
|---|---|
| `schema_meta` | Schema 版本和启动探针 |
| `episodes` | 请求、阶段、deadline、attempt 和最终结果 |
| `dispatches` | Worker 派发身份、lease、token、epoch 和接受状态 |
| `report_idempotency` | Worker 结果上报幂等记录 |
| `terminal_outcomes` | 已完成、取消、超时和迟到结果的稳定响应 |
| `agent_jobs` | AgentJob 的 pending、leased 和 completed 状态 |
| `gateway_sessions` | Gateway session 生命周期和清理状态 |
| `outbox` | 已持久化但尚未完成发布的结果事件 |

任务状态库与轨迹库相互独立：

- `server-state.db`：用于调度状态恢复；
- `trajectory.db`：用于轨迹元数据和查询索引；
- 轨迹大块 body 仍保存在轨迹目录的 `bodies/` 文件中。

## 4. 配置变化

### 4.1 Server 配置

`config/server.yaml` 新增并默认开启：

```yaml
persistence:
  enabled: true
  db_path: /home/uenv/server-state/server-state.db
  busy_timeout_ms: 5000
  terminal_ttl_secs: 86400
  idempotency_ttl_secs: 3600
  recovery_grace_secs: 30
  max_completed_entries: 100000
  max_result_bytes: 16777216
  max_database_bytes: 107374182400
  max_wal_bytes: 1073741824
  min_free_space_bytes: 10737418240
  writer_queue_capacity: 4096
  shutdown_grace_secs: 45
```

生产部署必须使用 worktree 之外的绝对路径，并保证目录权限和磁盘空间满足要求。

### 4.2 Worker 配置

`config/uenv-worker.yaml` 明确增加持久化 WAL 目录：

```yaml
wal:
  dir: /var/lib/uenv/wal
```

该目录必须位于可跨 Worker 进程重启保留的磁盘上。

## 8. 当前边界与后续建议

1. **已提交，尚未部署**  
   当前实现已提交到 `bridge-alignment` 分支；线上二进制仍未包含这些功能。

2. **SQLite 文件高水位**  
   后台会删除过期记录，但 SQLite `DELETE` 不保证物理文件立即缩小。建议后续开启 incremental auto-vacuum，并增加低频 `PRAGMA incremental_vacuum`。

3. **过期非终态对账**  
   当前运行时 deadline 和启动恢复会形成超时终态。建议再增加独立 reconciler，周期扫描超过 deadline 的非终态任务并报警。

4. **Worker 与 Server 同时重启的顺序**  
   Worker 首次启动时如果 Server 尚未监听，当前 CLI 可能因初始 transport error 退出。生产编排应保证 Server 先启动，或由 systemd/Kubernetes 重启策略重新拉起 Worker。

5. **单 Server 设计**  
   当前 SQLite 方案适合单机单进程 Server。若未来运行多个 Server 实例，需要把 `PersistenceStore` 后端替换为支持共享锁和集群一致性的数据库。

6. **监控建议**  
   应重点监控 persistence healthy、DB/WAL 大小、剩余空间、pending outbox、Worker WAL 数量和超过 deadline 的非终态任务数量。

## 9. 总结

本次改造实现了三层可靠性：

1. Server 使用 SQLite 保存任务状态机和终态；
2. Worker 使用 WAL 保存尚未确认的结果；
3. episode ID、request checksum、attempt、lease、token、epoch 和 idempotency key 共同保证重复通信不会变成重复执行。

因此，在单 Server 部署模式下，Server 重启后可以继续调度排队任务、接收执行中任务的迟到结果、返回已完成任务的持久化结果，并对异常或重复上报给出稳定响应。
