## 0. 概述

**UEnv-Hub = 内网「环境预缓存 + 制品分发」中心**（类比 Docker Hub / HF Hub，但同时托管 **manifest + 制品字节**）。

- 它**不参与 Episode 运行时**，只在**部署/扩缩容期**把环境备齐给 Worker；
- 目标是让 Worker 运行时**零外拉（zero egress）**——不访问任何公网/第三方 registry；
- 所有环境遵循**统一标准**：统一接口契约、统一配置契约、内容寻址的制品分发。

```
[外网导入机 一次性]  docker save / 编译 / 打包  ──┐
                                                  ▼ publish / publish-image
                               ┌──────────────────────────┐
                               │   uenv-hub（内网）        │  manifest + 制品字节（sha256）
                               └───────────┬──────────────┘
                                           │ uenv env sync（部署期，一次性）
                                           ▼
                               ┌──────────────────────────┐
                               │   uenv-worker            │  本地缓存；运行时只读，零外拉
                               └──────────────────────────┘
```

## 1. 标准化

### 1.1 统一的环境接口契约（OpenEnv 对齐）

每个环境/包都携带 **`interface`**：`action` / `observation` / `state` 三段 **JSON Schema**，对齐业界 [OpenEnv](https://github.com/huggingface/OpenEnv) 的 `models.py`。任何消费者（训练/Bridge）都能在**不看源码**的情况下，通过一个只读端点拿到环境的输入/输出/状态契约：

```
GET /api/v1/envs/{env_type}/versions/{version}/interface
GET /api/v1/packages/{package_id}/versions/{version}/interface
```

发布时对三段 schema 做 JSON Schema 合法性校验，不合规直接 `422` 拒绝。

### 1.2 统一的配置契约（`config_schema` + `default_config`）

每个环境用 **`config_schema`** 声明可接受的配置字段（类型、枚举、必填），并给出 **`default_config`**；发布时校验 `default_config` 必须满足 `config_schema`。这让「一个环境支持哪些参数/取值」成为**可查询的机器契约**，而不是口口相传。例如某些环境会用 `config_schema.properties.dataset.enum` 显式声明支持的数据集取值，供上层做路由对齐。

### 1.3 内容寻址的制品分发（两条通道）

| 机制 | 分发什么 | API / CLI | 典型场景 |
|------|----------|-----------|----------|
| **env registry**（轻量） | 环境 **manifest**：`interface` + `config_schema` + 依赖/资源等元数据 | `GET /api/v1/envs/...`；`uenv env publish` | 进程型/元数据契约 |
| **EnvPackage**（重制品） | **镜像 tar、catalog、数据包、依赖、worker overlay** | `GET /api/v1/packages/...`；`uenv env sync` / `publish-image` | 容器镜像、大数据/依赖包 |

EnvPackage 的每个 artifact **边写边算 sha256**、内容寻址，保证「Hub 存的 = Worker 拿的」可校验、可复现。

### 1.4 零外拉部署模型（air-gap ready）

- Worker 镜像拉取策略**默认 `local_only`**（不外拉），镜像字节由 Hub 以 `image_tar` 托管，Worker `docker load` 本地加载；
- 部署铁律三步：**导入机（有网，一次性）→ Hub（内网，publish）→ Worker（内网，sync）**；
- 支持**完全断网**：仓库内 `scripts/airgap-*.sh` 做离线编译包与镜像离线包（见 §6）。

## 2. 使用流程主线

### 2.1 连接 Hub + 鉴权

内网 Hub 开启单一共享 Bearer Token（`/api/v1/**` 全需 token；`/healthz`、`/version`、`/metrics` 公开）。

```bash
# 取 CLI 二进制（Hub 主机）
cd /root/uenv/uenv-hub && ~/.cargo/bin/cargo build --release -p uenv-hub-client
export PATH="/root/uenv/uenv-hub/target/release:$PATH"

# 持久化登录（推荐）；token 存放于 Hub 主机 /root/uenv/uenv-hub/data/.admin_token
uenv hub login --endpoint http://<HUB_IP>:8088 \
  --token "$(cat /root/uenv/uenv-hub/data/.admin_token)"

# 或环境变量（CI/临时）
export UENV_HUB_ENDPOINT=http://<HUB_IP>:8088
export UENV_HUB_TOKEN=uenvh_xxxxxxxx

uenv hub status          # -> reachable (N environments)
```

> 联调机示例：`http://8.130.95.176:8088`（同 VPC 内网 `http://192.168.0.133:8088`）。正式部署替换为你的内网地址。

### 2.2 查询标准化环境

```bash
uenv env list                    # 列出环境
uenv env versions <env>          # 列出版本（yanked 标记）
uenv env info <env>              # 完整 manifest（JSON）

TOKEN=$(cat /root/uenv/uenv-hub/data/.admin_token); HUB=http://<HUB_IP>:8088
# 读接口契约（Action/Observation/State）
curl -s -H "Authorization: Bearer $TOKEN" \
  "$HUB/api/v1/envs/<env>/versions/latest/interface" | python3 -m json.tool
# 读配置契约（可接受字段/枚举/默认）
curl -s -H "Authorization: Bearer $TOKEN" \
  "$HUB/api/v1/envs/<env>/versions/latest" \
  | python3 -c "import sys,json;d=json.load(sys.stdin);print(json.dumps(d.get('config_schema'),ensure_ascii=False,indent=2))"
```

### 2.3 发布环境（发布者/运维）

**env registry（元数据契约）**——脚手架 → 本地校验 → 发布：

```bash
uenv env init myenv --template math    # 模板: echo/math/code/agent
cd myenv
# 编辑 manifest.toml：env_type / [version] / [interface] / config_schema ...
uenv env validate                      # 纯本地校验，不连网、不需 token
uenv env publish                       # 发布 manifest（环境不存在会自动创建）
```

**EnvPackage（镜像/制品）**——导入机备料 → Hub 入库：

```bash
# 导入机（有网，一次性）
docker save <image> -o <image>.tar

# Hub 主机（内网）：镜像 tar 流式入库（sha256 边写边算）
uenv env publish-image <package_id> <version> --tar <image>.tar
# 其它 artifact（catalog / 数据包 / 依赖）作为 EnvPackage 制品 POST 入库，见标准化指南 §4
```

### 2.4 同步到 Worker（部署期，一次性）

```bash
# 拉 EnvPackage 全量制品到本地，并本地 docker load（运行时零外拉）
uenv env sync <package_id> --version <version> --docker-load
# 落盘 /var/lib/uenv/envs/<package_id>/<version>/（catalog / images.manifest / eval_spec / overlay / images/*.tar）

# env registry 元数据增量同步（不含大制品字节）
uenv hub sync --since <unix秒> [--dry-run]
```

## 3. 环境示例（标准化如何落到具体环境）

标准化对任意 `env_type` 一视同仁；下面仅作**用法示例**，不是手册重点：

| 环境 | 分发机制 | 标准化体现 |
|------|----------|-----------|
| `math`（进程插件） | env registry | 携带 `MathAction/Observation/State` 接口；`config_schema.dataset.enum` 声明支持的数据集取值 |
| `code`（进程插件） | env registry / EnvPackage | 携带 `CodeAction/Observation/State`；执行字段（`task_id/test_code/...`）在 `config_schema` |
| `swe`（容器） | EnvPackage | catalog + **image_tar** + eval_spec + overlay；接口 `SweAction(exec/write_file/apply_patch/submit)/Observation/State` |

> 消费者只需按 §2.2 读该环境的 `interface` 与 `config_schema`，即可正确组装请求，无需了解内部实现。

### 3.1 五类 Benchmark 在标准化 Hub 上的落地

当前五类 benchmark 收敛到 3 个 `env_type`，全部沿用上述标准化契约（接口 + 配置 + 内容寻址分发 + 零外拉）。消费者按 `dataset`（math/code 的路由键）或 `benchmark_variant`（swe）区分具体任务：

| Benchmark | `env_type` | 路由键 | Hub 分发机制 | Hub 上应缓存的制品 | Worker sync 后本地路径（示例） |
|-----------|-----------|--------|--------------|--------------------|--------------------------------|
| **PubMedQA** | `math` | `dataset=pubmedqa` | env registry | math manifest（+ 可选样本 tar） | `/var/lib/uenv/envs/math/0.2.0/` 或 `plugins/math/` |
| **SciTab** | `math` | `dataset=scitab` | env registry | 同上（+ 可选表格样本 tar） | 同上 |
| **OlymMATH** | `math` | `dataset=olymmath` / `-easy` / `-hard` | env registry | 同上（+ 可选题目 tar） | 同上 |
| **DSCodeBench** | `code` | `dataset=dscodebench` | EnvPackage | benchmark 树 + Python 依赖 + code 插件 + eval 脚本 | `/var/lib/uenv/envs/dscodebench/<v>/benchmark/`（→ `UENV_DSCODEBENCH_ROOT`） |
| **SWE-bench-Pro** | `swe` | `benchmark_variant=pro` | EnvPackage | catalog + **image_tar** + eval_spec + worker overlay | `/var/lib/uenv/envs/swe-bench-pro/0.2.0/` |

标准化 seed 已内置的 registry 契约（全新内网 Hub 首启自动写入，幂等增量）：

- **`math@0.2.0`**：`config_schema.dataset.enum = [gsm8k, pubmedqa, scitab, olymmath, olymmath-easy, olymmath-hard]`，对齐 `plugins/math/manifest.yaml` 与 Worker `payload.rs` / `plugins/math/src/score.rs` 的路由键；
- **`code@0.2.0`**：`config_schema.dataset.enum = [dscodebench]`，执行字段（`task_id`/`test_code`/`test_script_path`/`library`/`num_tests` 等）在 `config_schema`。

按 benchmark 消费/发布的具体命令：

```bash
# 消费者：确认某 env 支持哪些 dataset（对齐上层路由）
curl -s -H "Authorization: Bearer $TOKEN" "$HUB/api/v1/envs/math/versions/latest" \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['config_schema']['properties']['dataset']['enum'])"
# -> ['gsm8k','pubmedqa','scitab','olymmath','olymmath-easy','olymmath-hard']

# 运维：DSCodeBench 全量（EnvPackage）
uenv env sync dscodebench --version 0.1.0
export UENV_DSCODEBENCH_ROOT=/var/lib/uenv/envs/dscodebench/0.1.0/benchmark

# 运维：SWE-bench-Pro 镜像 tar + catalog（零外拉）
uenv env publish-image swe-bench-pro 0.2.0 --tar <instance>.tar   # Hub 主机
uenv env sync swe-bench-pro --version 0.2.0 --docker-load          # Worker
```

> 说明：`math`/`code` 的 registry 契约随 seed 自动就绪；**DSCodeBench 全量 benchmark、SWE-Pro 真实镜像 tar 属导入机侧真实文件**，需按「导入机 → `publish`/`publish-image` → Worker `sync`」备齐，不随源码内置。判分逻辑均在 Worker 侧（swe/math 已成熟，DSCodeBench 为 MVP）。横向支持矩阵详见 [五类 Benchmark 跨层调整](../五类Benchmark-Worker支持现状与跨层调整.md)。

## 4. 关键部署变量

| 变量 | 说明 |
|------|------|
| `UENV_HUB_ENDPOINT` / `UENV_HUB_TOKEN` | 内网 Hub 地址与共享 token |
| `UENV_SWE_IMAGE_PULL_POLICY` | 容器环境镜像策略，**默认 `local_only`（零外拉）** |
| `swe.env_package_dir` | 容器环境：Hub sync 后的 EnvPackage 本地目录 |
| `UENV_HUB_SWE_IMAGE_DIR` | Hub 主机上镜像 tar 暂存目录（seed 自动托管） |
| `UENV_MATH_PLUGIN_BIN` | math 全系列：指向 `uenv-math-plugin` |
| `UENV_CODE_PLUGIN_BIN` / `UENV_CODE_EVAL_SCRIPT` | DSCodeBench：code 插件 / 评测脚本（默认 `plugins/code/scripts/evaluate_code.py`） |
| `UENV_DSCODEBENCH_ROOT` | DSCodeBench 全量：Hub sync 后的 benchmark 根目录 |

### 5.1 本地全量测试（开发/CI）

```bash
cd uenv/uenv-hub
cargo test -p uenv-hub-core -p uenv-hub-server -p uenv-hub-types

#       镜像 tar 托管与流式下发、零外拉默认策略、发布校验等。
```

### 5.2 真机非破坏式冒烟测试

在运行中的生产 Hub 上，用一次性环境验证「标准化契约往返 + 鉴权 + 发布校验」，再软删除清理（生产环境全程不动）：

```bash
TOKEN=$(cat /root/uenv/uenv-hub/data/.admin_token); HUB=http://<HUB_IP>:8088; ENV=std-probe-$$
AUTH="Authorization: Bearer $TOKEN"; CT="Content-Type: application/json"
curl -s -X POST "$HUB/api/v1/envs" -H "$AUTH" -H "$CT" -d "{\"env_type\":\"$ENV\",\"tags\":[\"probe\"]}" >/dev/null
curl -s -X POST "$HUB/api/v1/envs/$ENV/versions" -H "$AUTH" -H "$CT" -d '{
  "version":"0.2.0",
  "interface":{"action":{"title":"DemoAction","type":"object","properties":{"response_text":{"type":"string"}}}},
  "config_schema":{"type":"object","properties":{"mode":{"type":"string","enum":["a","b"]}},"required":["mode"]},
  "default_config":{"mode":"a"}
}' >/dev/null
curl -s -H "$AUTH" "$HUB/api/v1/envs/$ENV/versions/0.2.0/interface" | python3 -m json.tool   # 应返回 action
curl -s -X DELETE "$HUB/api/v1/envs/$ENV" -H "$AUTH" -o /dev/null -w "cleanup=%{http_code}\n"  # 204
```

### 5.3 实机联调结论（2026-07-10，`8.130.95.176:8088`）

| 检查项 | 结果 |
|--------|------|
| 公开端点 `/healthz` / `/version` | `200` / `uenv-hub 0.1.0` ✅ |
| 鉴权：无 token 访问 `/api/v1/**` | `401 UNAUTHORIZED` ✅ |
| 环境列表（带 token） | `agent/code/math` ✅ |
| 发布标准化 manifest（create/publish） | `201 / 201` ✅ |
| 接口契约往返（`GET version` + `GET .../interface`） | `action/observation/state` 三段齐全 ✅ |
| 配置契约往返（`config_schema` + `default_config`） | 往返一致、枚举保留 ✅ |
| 发布校验：非法 interface schema | `422`（正确拒绝）✅ |
| 清理：软删除 → 复查 | `204 → 404` ✅ |
| 生产环境 `agent/code/math` | 全程未受影响 ✅ |

> 结论：**实机联调通过**。生产 Hub 进程未动、数据未污染。

## 6. 完全断网（air-gap）：两步一次性准备

```bash
# 准备① 离线编译源码包（有网机器）
scripts/airgap-offline-bundle.sh            # 产出 dist/uenv-hub-offline-src-<日期>.tar.gz
#   搬进内网后：解压 → bash scripts/airgap-offline-build.sh（cargo build --offline，不联网）

# 准备② 镜像离线包（有网机器）
scripts/airgap-image-bundle.sh <image> ...  # 产出 dist/uenv-hub-images-<日期>.tar.gz
#   搬进内网 Hub 后：uenv env publish-image ... --tar images/*.tar
```

## 8. 常见错误码 / FAQ

| 现象 | 原因 / 处理 |
|------|-------------|
| `401 UNAUTHORIZED` | 缺 token 或失效；带 `Authorization: Bearer <token>`（`/api/v1/**` 全需） |
| `409 VersionAlreadyExists` | 同版本已发布；换版本号或先 yank |
| `422 InvalidManifest / SchemaValidationFailed` | `interface`/`config_schema` 不是合法 JSON Schema，或 `default_config` 不满足 `config_schema` |
| `env sync` 后 Worker 仍外拉镜像 | 未 `--docker-load`，或未保持 `UENV_SWE_IMAGE_PULL_POLICY=local_only`（默认已是） |
| `latest` 不是期望版本 | `latest` 取**最高 semver**（非最近发布）；与旧占位版本并存时需 yank 旧版本 |

## 9. 术语 / 参考

- **env registry**：Hub 的经典环境元数据分发（manifest + interface + config），轻量、无大文件字节。
- **EnvPackage**：Hub 的重制品分发单元（镜像 tar / catalog / 数据包 / overlay），内容寻址（sha256）。
- **interface（OpenEnv 契约）**：Action / Observation / State 三段 JSON Schema。
- **config 契约**：`config_schema` + `default_config`，声明并校验环境可接受的配置。
- **zero egress（零外拉）**：Worker 运行时不访问任何公网/第三方 registry；镜像默认 `local_only`。
