## 2. MVP 阶段定义

**MVP = 实现规划中的 P0**，目标：用户能 **开始一轮训练 → 实时看到链路状态 → 本地抓拍快照 → 终止训练**，数据来自真实模块上报而非纯 mock。

### 2.1 MVP 必须交付（三人共同验收）

| # | 能力 | 验收方式 |
|---|------|----------|
| M1 | 用户通过前端 start 一轮 run，获得 `training_run_id` 并建立 SSE | 顶栏状态变为 `RUNNING`，SSE 连接指示为已连接 |
| M2 | Adapter/Server/Worker 关键事件经聚合层归并后推送到前端 | 工作流节点随 episode 推进变化（submit → dispatch → execute → done） |
| M3 | 树状视图展示 run → worker → episode → step 层级 | 至少 1 个完整 episode 可在树中展开到 step |
| M4 | 用户可随时抓拍快照，切到快照视图查看静态链路，再切回实时 | 抓拍不中断 SSE；快照内容与抓拍时刻一致 |
| M5 | 用户 stop run，run 进入关闭边界，后续僵尸事件不回写 UI | 顶栏 `CLOSED`；迟到事件进 `late_events` 表 |
| M6 | SSE 断线后自动重连，状态可恢复 | 断网 10s 后恢复，UI 与断线前一致或经 `full_state` 补齐 |
| M7 | 聚合层重启后可从 SQLite 恢复当前态 | 重启聚合层后前端重连，状态不丢 |

### 2.2 MVP 可选（有余力再做，不阻塞验收）

| # | 能力 | 负责人 |
|---|------|--------|
| O1 | 工作流节点与树节点选中联动 | B |
| O2 | 顶栏展示 `updated_at` / 数据滞后提示 | B |
| O3 | run 内按 id、状态搜索/过滤 | B |
| O4 | 快照持久化至 localStorage | B |
| O5 | 高频 step 批处理 flush window（100–250ms） | A |

### 2.3 MVP 明确不包含

- 日志 / Metrics 独立 Tab（P1）
- 历史回放 API 与回放 UI（P1）
- 侧边多快照面板（P2）
- GreptimeDB metrics 管道（P2）

## 7. MVP 模块事件最小集

> C 在 S0-5 中扩展为完整对照表；MVP 至少保证以下事件可上报且被 A 正确归并。

| 模块 | MVP 必须实现的 `event_type` |
|------|----------------------------|
| Adapter | `RUN_STARTED`, `RUN_STOPPED`, `EPISODE_SUBMITTED` |
| Server | `EPISODE_DISPATCHED`, `ATTEMPT_STARTED`, `EPISODE_CLOSED`, `RUN_CLOSED` |
| Worker | `STEP_STARTED`, `STEP_COMPLETE`, `EPISODE_COMPLETED`, `WORKER_HEARTBEAT` |

MVP 可延后：`EPISODE_FAILED`, `ATTEMPT_CLOSED`, `WORKER_REGISTERED`, `HEARTBEAT`（通用）

## 11. 风险与依赖

| 风险 | 影响 | 缓解 |
|------|------|------|
| 契约冻结延迟 | 三人返工 | W0 必须评审签字；字段名不得私自改 |
