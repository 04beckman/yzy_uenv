# UEnv 前端可视化讨论稿

> 目标：为当前项目补一个可视化前端，用来观察单次测试/训练链路的整体状态、模块关系和实时进展。
>
> 说明：本文是设计讨论稿，不是实现方案。第 7 节架构决策已冻结，后续实现应以此为准；其余章节仍保留讨论背景与取舍依据。

## 1. 我对两种模式的判断

我认为你提出的两种模式是合理的，而且是互补的，不是互斥的。

- **快照模式**适合回答“现在链路到底走到哪了”
- **流式模式**适合回答“链路是怎么一路走过来的”

这两个问题本质不同：

- 快照强调“当前状态的可读性”
- 流式强调“状态变化的连续性”

如果只做流式，更新太快时很容易错过关键步骤；如果只做快照，又会丢掉中间过程和时序关系。对训练链路这种多模块、强异步场景来说，双模式是更稳妥的设计。

## 2. 推荐结论

我的建议不是把它们做成两个完全割裂的入口，而是做成同一个页面里的两种查看方式：

- 默认进入 **流式模式**
- 提供 **“抓取快照”** 和 **“冻结当前流”** 的能力
- 允许用户在流式视图里一键切到某个快照时刻查看细节

原因很简单：

- 流式更符合日常使用场景，用户通常想先看“正在发生什么”
- 快照更像一个“暂停点”或“当前帧”，适合定位问题
- 如果把快照做成完全独立模式，切换成本会比较高，操作也更绕

我更倾向于：

- **默认流式**
- **旁边提供快照按钮**
- **快照可固定保存多个版本**
- **流式状态可暂停/恢复**

这样用户既能看实时进展，也能保留几个关键时刻用于对比。

## 3. 数据通道怎么做

这里我不建议把“日志”直接当成前端主数据源。

### 3.1 不建议只靠日志

日志的问题是：

- 面向人，不面向状态机
- 不稳定，格式容易漂
- 很难保证顺序、幂等、去重
- 前端想做结构化展示时，解析成本很高

日志可以保留，但应该是辅助诊断，而不是 UI 的核心输入。

### 3.2 推荐的主数据源

前端应该消费的是**结构化事件**，而不是原始日志文本。完整字段定义见 **§7.5**；概要包括：

- 排序与去重：`event_id`、`correlation_id`、`seq`、`timestamp`
- 来源与语义：`source_id`、`module`、`entity_type`、`entity_id`、`event_type`、`schema_version`
- 业务索引：`episode_id`、`attempt_id`、`worker_id` 等（见 §7.4）
- 载荷：`payload`

这样前端才能稳定地做：

- 进度条更新
- 树结构展开/收起
- 节点状态高亮
- 异步乱序处理
- 历史回放

### 3.3 推荐的传递方式

结合你们当前架构，我更推荐：

- **模块内部仍然走现有协议**
  - Worker 对 Server 继续用 `DispatchEpisode` 的 `StreamReport`
  - Worker 对控制面继续用心跳/注册/结果上报
- **再增加一层面向可视化的聚合层**
  - 把各模块上来的结构化事件统一收敛
  - 对前端提供 snapshot 和 stream 两类接口

也就是说，前端不直接盯着每个模块的原始输出，而是看一个“状态汇聚后的视图”。

### 3.4 技术选择建议

如果你问我技术怎么选，我的建议是：

- **首选：Server 侧聚合 + SSE/WebSocket 推送到前端**
- **Worker/Server 间：继续用现有 gRPC / stream 报告**
- **持久化：先做事件落盘或数据库存储，不要一开始就把 Redis / GreptimeDB 当成主通道**

我的判断：

- `Redis` 适合做缓存、订阅广播、临时状态，但不适合作为唯一事实源
- `GreptimeDB` 更偏时序分析和查询，不适合直接承担前端实时状态编排
- 如果只为前端实时展示，先把“事件模型”和“状态聚合”做好，比先选数据库更关键

### 3.5 异步更新与通信策略

异步更新一定会发生；通信策略已冻结为 **事件驱动为主、定时心跳为辅**（详见 §7.3）。

规则：

- 关键状态变化时主动产出进展事件；心跳仅用于存活探测与断链补发
- 每条事件都带 `seq` 与 `event_id`
- 前端只接受“更晚”的状态，不接受倒退
- 同一 `episode_id` 或同一 `worker_id` 的更新要能幂等覆盖
- 对超时、断线、重试，要有明确的 `attempt_id`
- 对于当前不完整的状态，UI 要显示“最后更新时间”和“数据可能滞后”

所有结构化进展事件按时间追加保存，供历史回放（§7.7）；**流式更新不自动产生快照**，快照仅用户手动抓拍（§7.6）。

## 4. 快照模式和流式模式怎么并存

我建议不要把它们做成完全等价的两个页面，而是做成以下关系：

- **流式模式是主视图**
- **快照是流式视图上的一个可冻结状态**
- **快照可单独保存为检查点**

这样比“纯切模式”更顺手。

### 4.1 方案 A：严格模式切换

用户明确选择“快照”或“流式”，两套渲染逻辑完全分开。

优点：

- 心智模型简单
- 实现也直观

缺点：

- 用户容易丢上下文
- 频繁切换比较烦
- 不适合实时跟踪和定位同时存在的场景

### 4.2 方案 B：默认流式 + 快照按钮

用户先看流式，必要时一键抓取当前状态作为快照。

优点：

- 更贴近日常使用
- 保留实时感
- 快照天然就是“关键时刻”

缺点：

- 需要一个聚合层来保证快照一致性
- UI 交互要处理“冻结态”和“实时态”的关系

### 4.3 方案 C：流式主界面 + 侧边快照面板

主区展示实时链路，侧边栏列出多个快照点，用户可以随时切换查看。

优点：

- 最适合调试和复盘
- 可同时保留“当前”和“历史”

缺点：

- UI 复杂度最高
- 初版实现成本更高

### 4.4 我的推荐

我推荐 **方案 B 作为第一版**，后续再演进到 C。

理由：

- 先把“实时链路看得见”做出来最重要
- 快照本质上是对实时状态做一次固化，先有实时态，后面再扩展历史态更自然

## 5. 展示形式怎么选

你提到的两个方向都对，但它们解决的问题不同。

### 5.1 工作流式 UI

适合展示：

- episode 从提交到完成的整体路径
- 当前执行到了哪一步
- 哪个模块正在工作
- 哪个环节卡住了

优点：

- 强调时序和进展
- 对训练/调度链路非常直观
- 很适合“实时看链路”

缺点：

- 不擅长表达层级关系
- 很多节点展开后会显得拥挤

### 5.2 树状图

适合展示：

- worker 有几个
- 每个 worker 上有几个环境实例
- 每个实例下面有几个 episode
- 每个 episode 当前第几个 step

优点：

- 层次关系清晰
- 容易做筛选和 drill-down

缺点：

- 不太能表现“过程流动”
- 很容易变成纯状态列表

### 5.3 我的建议

不要二选一，建议做成 **工作流主视图 + 树状详情视图**：

- **主视图用工作流**
  - 看链路进展
  - 看当前卡点
  - 看状态变化
- **侧边栏/抽屉用树状图**
  - 看 worker / instance / episode / step 的层级
  - 看当前节点的明细

如果只能先做一种，我建议先做 **工作流式 UI**，因为它更贴合你的实时追踪目标。树状图可以作为第二阶段补齐。

## 6. 我建议的最小可行方案

如果目标是尽快出一个能用的前端，第一版建议：

1. 独立**聚合层**接入各模块事件；支持按 §7.4 索引字段（至少 `correlation_id` / `episode_id`）查询当前状态
2. 订阅同一条链路的结构化事件流（事件驱动更新 + 心跳补发）
3. 主视图用 workflow 展示进度；右侧用 tree 展示 worker / env / episode / step 关系
4. 提供 **手动抓拍快照**（纯前端）、**快照 ↔ 实时切换**（详见 [实现规划 §3](./260612-frontend-visualization-implementation-plan.md)）
5. 日志 / metrics 独立面板，共用 `correlation_id` 上下文切换（§7.8）
6. 事件持久化与回放数据模型就绪；回放 UI 可分期交付（§7.7）

## 7. 架构决策（已冻结）

以下决策已确认，后续接口设计与实现应以此为准。

### 7.1 事实源：独立聚合层

**决策**：新增独立的**可视化聚合层**，作为前端展示的事实源；各业务模块（Adapter、Server、Worker 等）向聚合层汇总结构化事件，聚合层负责收集、去重、排序、归并状态并对外提供查询与推送。

**边界划分**：

- **Server 保留调度事实**——episode 如何分发、租约、重试等控制面语义仍由 Server 负责
- **聚合层负责展示事实**——前端消费的 snapshot / stream 均来自聚合层，不直接绑定 Server 转发链路

聚合层是**状态投影器**，不是同步代理：模块可独立向上报送事件，不必强依赖 Server 中转。

**单点风险**：聚合层挂掉时，训练与调度照常进行，仅 UI 暂时不可见；风险等级低于“Server 兼做唯一观测中转”的方案。

### 7.2 上报链路韧性

**决策**：采用文档 7.3 讨论的策略，避免“上报链路断掉就全断”。

1. **模块本地先记账**——Worker / Server / Adapter 将关键事件写入本地缓冲或 WAL，断链不丢历史
2. **异步上报到聚合层**——重试、批量发送、幂等写入；聚合层去重、排序、归并
3. **控制面与观测面分离**——观测上报失败不阻断调度与 episode 执行
4. **允许补发**——链路恢复后补发缺失事件；聚合层按 `event_id` / `seq` 去重后补齐状态

聚合层不得要求“每条事件同步确认后才能继续执行业务”。

### 7.3 通信更新策略：事件驱动为主，心跳为辅

**决策**：采用**事件驱动为主、定时心跳为辅**的通信更新策略。

| 机制 | 职责 |
|------|------|
| **进展事件（主）** | 在关键状态变化时主动产出结构化事件（及对应日志），向上报送聚合层；驱动流式 UI 更新与事件日志追加 |
| **定时心跳（辅）** | 证明模块与上报链路存活；携带轻量存活/滞后信号；链路恢复时触发 WAL 补发与状态对齐 |

**关键状态变化**包括但不限于：episode 提交、调度分发、step 完成、结果回报、失败/重试、worker 注册/注销等。

心跳**不替代**进展事件；心跳主要用于存活探测与断链恢复，不作为 UI 主更新源。

### 7.4 索引字段体系

**决策**：索引字段按下列清单冻结；数据模型须支持**从任意一个字段出发，关联并还原整条链路**（正向 drill-down 与反向定位均可用）。

#### 7.4.1 业务索引字段（层级由粗到细）

| 字段 | 类型 | 说明 | 典型来源 |
|------|------|------|----------|
| `training_run_id` | string | 一次训练/run 的唯一标识 | Adapter / 训练框架 |
| `adapter_run_id` | string | Adapter 侧 run 标识；可与 `training_run_id` 互映射 | Adapter |
| `batch_id` | string | 批次标识（如 VeRL batch） | Adapter |
| `correlation_id` | string | 全链路 trace，单样本/单 episode 级 | Adapter → Server → Worker |
| `episode_id` | string | Episode 唯一 ID | Server / Worker |
| `attempt_id` | uint32 | 同一 episode 的重试序号，从 1 起 | Server |
| `worker_id` | string | 执行 Worker | Server / Worker |
| `env_instance_id` | string | Worker 内环境实例 | Worker |
| `step_index` | int32 | Episode 内步序号 | Worker |

#### 7.4.2 辅助关联字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `dispatch_lease_id` | string | 调度租约，failover 场景定位 |
| `scheduler_epoch` | uint64 | 调度器 epoch，区分调度代际 |
| `env_type` | string | 环境类型（如 `math`） |
| `source_id` | string | 事件来源模块/实例（见 7.5） |

#### 7.4.3 时间索引

| 字段 | 类型 | 说明 |
|------|------|------|
| `timestamp` | int64 | 事件发生时间（Unix 毫秒）；跨源排序与回放时间轴 |
| `seq` | uint64 | 同一 `source_id` 内单调递增序号 |

#### 7.4.4 关联规则

- 每条事件、每条日志、每个 metric 样本**至少携带** `correlation_id`；能填写的业务索引字段应尽量齐全
- 聚合层维护**多键反向索引**（如 `episode_id → correlation_id`、`worker_id → [episode_id…]`），保证从任意已知字段可跳转至同链路其他层级
- 层级语义：`training_run_id` ⊃ `batch_id` ⊃ `correlation_id` ⊃ `episode_id` ⊃ `step_index`；`worker_id` / `env_instance_id` 为执行侧正交维度

### 7.5 事件模型：完整骨架

**决策**：事件模型采用覆盖最完善的骨架，在排序三元组之上扩展业务与演进字段。

#### 7.5.1 排序与去重（核心三元组 + 唯一标识）

| 字段 | 必填 | 说明 |
|------|------|------|
| `event_id` | 是 | 全局唯一，幂等去重 |
| `correlation_id` | 是 | 串联同一条链路 |
| `seq` | 是 | 同一 `source_id` 内单调递增 |
| `timestamp` | 是 | Unix 毫秒；跨源对齐与回放时间轴 |

#### 7.5.2 来源与语义

| 字段 | 必填 | 说明 |
|------|------|------|
| `source_id` | 是 | 事件来源（如 `adapter:verl-1`、`server:sched-1`、`worker:5e96910f`） |
| `module` | 是 | 逻辑模块名（`adapter` / `server` / `worker` / `env`） |
| `entity_type` | 是 | 事件作用实体类型（`training_run` / `episode` / `worker` / `env_instance` / `step`） |
| `entity_id` | 是 | 与 `entity_type` 对应的实体 ID |
| `event_type` | 是 | 事件语义（如 `EPISODE_SUBMITTED`、`DISPATCHED`、`STEP_COMPLETE`、`HEARTBEAT`） |
| `schema_version` | 是 | 事件 schema 版本，兼容旧数据解析 |

#### 7.5.3 业务索引（事件内嵌，支持任意字段关联）

| 字段 | 必填 | 说明 |
|------|------|------|
| `training_run_id` | 否 | 有则填 |
| `adapter_run_id` | 否 | 有则填 |
| `batch_id` | 否 | 有则填 |
| `episode_id` | 否 | 有则填 |
| `attempt_id` | 否 | 有则填 |
| `worker_id` | 否 | 有则填 |
| `env_instance_id` | 否 | 有则填 |
| `step_index` | 否 | 有则填 |

#### 7.5.4 载荷

| 字段 | 必填 | 说明 |
|------|------|------|
| `payload` | 否 | JSON；阶段详情、错误信息、指标快照等 |

`event_type = HEARTBEAT` 时 `payload` 仅携带存活/滞后类轻量信息，不承载完整状态。

### 7.6 快照：状态 + 游标

**决策**：快照冻结为**某一时刻抓拍保存的历史记录**，形态为 **状态 + 游标**，不是页面截图。

| 组成部分 | 内容 |
|----------|------|
| **状态** | 抓拍时刻聚合层计算出的链路视图（workflow + tree 可渲染的结构化状态） |
| **游标** | 对应的事件位置（如 `event_id` / `(source_id, seq)` / 全局偏移），以及 `captured_at` 时间戳 |
| **索引** | 关联的 `correlation_id` 及 7.4 业务索引字段，便于从快照跳转整条链路 |

**查看行为**：

- 借助 7.5 事件模型，从游标向前重放事件，可串联并还原该时刻的**完整链路情况**（按 `timestamp` 排列）
- 用户抓拍后**可立即查看**该快照
- 抓拍**不中断、不替换**流式事件回传与缓存；用户可在**快照视图**与**实时流式视图**之间自由切换

**重要区分**：流式推送的每次事件更新**不会**自动产生快照；快照**仅由用户手动触发**（见 7.7）。

### 7.7 历史回放

**决策**：**需要历史回放**；数据模型与存储从第一版起按可回放设计（回放 UI 可分期交付）。

#### 7.7.1 更新与存储

- **回传/更新时机**：与 7.3 一致——进展事件驱动为主，心跳为辅
- **流式实时展示**：订阅聚合层事件流，事件到达即更新 UI
- **事件持久化**：所有结构化进展事件**追加落盘**（append-only event log），供回放重放
- **快照**：**仅用户手动抓拍**产生；自动事件更新**不得**转换为快照

#### 7.7.2 回放能力

- 按 `correlation_id`（或 7.4 任意索引字段）拉取事件日志，按 `timestamp` + `seq` 重放，重建任意时刻 UI 状态
- 手动快照可作为回放**书签/检查点**（从游标继续向前播放，无需每次从 t₀ 重算）
- 支持：完整过程复盘、时刻暂停、多 run 对比、追查被后续状态掩盖的短暂异常

#### 7.7.3 回放与快照的关系

| 能力 | 数据来源 | 触发方式 |
|------|----------|----------|
| 实时流式 | 事件流 + 当前聚合状态 | 自动（事件驱动） |
| 历史回放 | 持久化事件日志 + 状态重放器 | 用户进入回放模式 |
| 手动快照 | 状态 + 游标 | **仅用户手动抓拍** |

### 7.8 日志、metrics、事件流：分页面，共 `correlation_id`

**决策**：日志、metrics、事件流**不放在同一个页面**；在存储层共用 `correlation_id`（及 7.4 索引字段）绑定，在界面上通过 Tab/面板**切换查看**。

| 数据层 | 语义 | UI 位置 |
|--------|------|---------|
| **事件流** | 状态变化主线；驱动 workflow / 回放 | 主可视化页（工作流 + 树） |
| **日志** | 原始诊断，回答“为什么” | 独立日志面板/页，按 `correlation_id` 过滤 |
| **metrics** | 聚合指标，回答“系统压力如何” | 独立 metrics 面板/页，按 `correlation_id` 或 run 过滤 |

同一链路内切换三种视图时，**保持 `correlation_id` 上下文不变**；后端存储分表/分库，不做语义混用。

### 7.9 决策摘要

| # | 议题 | 冻结结论 |
|---|------|----------|
| 1 | 事实源 | 独立聚合层；各模块向聚合层汇总事件 |
| 2 | 链路韧性 | 本地 WAL + 异步上报 + 观测/控制分离 + 补发 |
| 3 | 通信策略 | 事件驱动为主，定时心跳为辅（存活与补发） |
| 4 | 索引 | 7.4 字段清单；任意字段可关联整条链路 |
| 5 | 事件模型 | 7.5 完整骨架 |
| 6 | 快照 | 状态 + 游标；手动抓拍；可即时查看；不影响实时流 |
| 7 | 历史回放 | 需要；事件追加存储；更新不自动转快照 |
| 8 | 日志/metrics/事件流 | 分页面展示；存储层共 `correlation_id` |

## 8. 当前结论

综合前文讨论与第 7 节冻结决策，当前结论如下：

- **两种模式都保留**：默认流式；快照为用户手动抓拍的历史记录（状态 + 游标）
- **主数据是结构化事件**，不是日志；通信上事件驱动为主、心跳为辅
- **展示以工作流为主，树状图为辅**；日志 / metrics / 事件流分页面，共用 `correlation_id`
- **独立聚合层**为前端事实源，不参与业务调度；上报链路断掉不阻断训练执行
- **历史回放**需要，事件模型与存储按可回放设计；流式更新不自动产生快照
