# UEnv 前端可视化页面详细设计方案

> **版本**：2026-06-12  
> **定位**：给前端页面实现和视觉设计直接使用的详细设计稿，描述布局、组件、交互、后端对接职责与页面状态，不涉及 HTML/CSS 代码  
> **参考**：[`260612-frontend-visualization-implementation-plan.md`](./260612-frontend-visualization-implementation-plan.md)

---

## 1. 设计目标

这个页面的目标不是做一个泛化仪表盘，而是做一个训练过程的操作台。用户打开后应该立刻看到三件事：

1. 当前 run 正在发生什么
2. 整条工作流推进到哪里了
3. 某个异常点为什么异常

因此，页面必须同时满足：

- 适合实时观察
- 适合快速定位
- 适合回看历史快照
- 适合从整体下钻到局部

---

## 2. 页面信息架构

### 2.1 主页面由四个区域组成

| 区域 | 位置 | 作用 | 默认占比 |
|------|------|------|----------|
| 顶部控制栏 | 页面最上方 | run 状态、模式切换、全局操作 | 约 10% 高度 |
| 主工作区 | 中间主体 | 工作流视图、树状图、详情面板 | 约 70% 高度 |
| 底部辅助区 | 页面底部 | 日志、指标、事件流、快照列表 | 约 20% 高度 |
| 全局浮层 | 页面上层 | 搜索结果、错误提示、确认弹窗 | 按需出现 |

### 2.2 主工作区建议采用三栏结构

中间主体建议按以下比例分布：

- 左侧 58%：工作流视图
- 中间分隔区 2%
- 右侧 40%：树状图 + 节点详情

如果窗口较窄，右侧面板可折叠为抽屉，但默认桌面布局必须保持双视图同时可见。

### 2.3 页面浏览顺序

用户进入页面后的自然阅读顺序应该是：

1. 顶部状态条
2. 左侧工作流视图
3. 右侧树状图
4. 右侧详情卡片
5. 底部日志与指标

这个顺序决定了信息层级，不能让日志或辅助区压过主流程。

## 3. 顶部控制栏设计

### 3.1 左侧区域：页面身份与当前运行对象

左侧建议放：

- 产品名或模块名
- 当前页面标题，例如“Training Run Visualization”
- 当前 `training_run_id`
- run 的简短状态标签，例如 `RUNNING` / `STOPPING` / `CLOSED`

#### 展示方式

- 标题使用大字号
- ID 使用等宽字体
- 状态标签使用高对比色块
- 如果当前为快照视图，应在标题旁显示“Snapshot”标记

### 3.2 中间区域：运行摘要

中间区域建议显示：

- 当前激活阶段
- 当前激活 episode 数
- 活跃 worker 数
- 最近更新时间
- SSE 连接状态

这部分不是装饰卡片，而是运行健康度摘要。  
建议只显示最关键的 4 到 5 项，不要把所有统计都塞进去。

### 3.3 右侧区域：全局操作

右侧按钮建议固定为：

- 开始运行
- 停止运行
- 抓取快照
- 刷新状态
- 搜索
- 模式切换

#### 按钮层级

- 主按钮：开始 / 停止
- 次按钮：抓取快照 / 刷新
- 工具按钮：搜索 / 模式切换

#### 行为要求

- 开始和停止必须二选一显示，避免同时出现造成误解
- 抓取快照成功后要立即显示快照名称或时间戳
- 切换模式时顶部必须明确提示当前是实时还是快照

## 4. 主工作区详细布局

## 4.1 左侧：工作流视图

工作流视图是主视觉核心，负责展示训练链路的阶段推进。

### 4.1.1 视图结构

建议采用自上而下或从左到右的阶段流，推荐横向展开，原因是更适合同时观察分支与状态变化。

#### 推荐结构

- 左到右排列阶段节点
- 主路径水平展开
- 分支和重试向下折线展开

### 4.1.2 节点构成

每个工作流节点建议包含以下元素：

- 节点标题
- 节点状态色
- 该节点关联的实体数量
- 最近更新时间
- 异常标记或延迟标记

### 4.1.3 节点视觉层级

节点的视觉结构建议分成三层：

1. 外框：表示类型或层级
2. 填充：表示状态
3. 内文：表示名称和摘要

#### 节点状态表现

| 状态 | 视觉表现 | 含义 |
|------|----------|------|
| PENDING | 低饱和灰色 | 尚未开始 |
| ACTIVE | 强调色高亮 + 轻微脉冲 | 正在进行 |
| DONE | 稳定成功色 | 已完成 |
| FAILED | 错误色 + 明显边框 | 已失败 |
| SKIPPED | 降低透明度 | 被跳过 |
| CLOSED | 中性关闭色 | 已关闭，不再变化 |

### 4.1.4 连线设计

连线用于表达阶段之间的传递关系。

- 主路径连线使用较粗线条
- 当前活跃路径连线使用高亮色
- 已完成路径连线使用稳定中性色
- 失败路径连线使用错误色

#### 分支表现

- 重试分支应向下偏移，避免和主路径重叠
- 异常分支必须保留方向箭头，不能省略
- 当节点被折叠时，连线应自动压缩，不影响主路径可读性

### 4.1.5 工作流视图内的局部组件

工作流视图内部建议包含：

- 缩放控制
- 适配窗口按钮
- 居中当前节点按钮
- 显示/隐藏辅助标记开关
- 搜索定位输入框

#### 缩放规则

- 默认 100%
- 可缩到看清全局结构
- 可放大到查看单个节点字段
- 缩放时节点布局必须保持稳定，不能跳位

### 4.1.6 工作流视图的空态与异常态

- 没有数据时显示“等待 run 数据进入”
- 连接断开时显示“实时链路已暂停”
- 当前 run 已结束时保留完整图，但弱化动态效果

---

## 4.2 右侧：树状图 + 节点详情

右侧区域建议垂直分为上下两块：

- 上半部分：树状图
- 下半部分：节点详情

### 4.2.1 树状图定位

树状图不是工作流的重复图，而是“对象层级导航器”。

它用于回答：

- run 下面有哪些 worker
- worker 下面有哪些 env instance
- env instance 下面有哪些 episode
- episode 下面有哪些 step

### 4.2.2 树状图的展示方式

建议使用可展开层级树，每一层的节点结构保持一致：

- 左侧折叠箭头
- 中间名称与 ID
- 右侧状态徽标
- 可选计数徽标

#### 树节点信息密度

每个树节点最多展示三类信息：

1. 名称
2. 状态
3. 数量或序号

更多信息统一放到详情区，不在树上堆叠。

### 4.2.3 树状图的交互

- 点击节点：选中并联动详情面板
- 点击折叠箭头：展开或收起子树
- 双击节点：快速跳转到工作流视图中对应位置
- 悬停节点：显示简短摘要

### 4.2.4 树状图的视觉规则

- 当前选中节点高亮
- 当前活跃节点额外标记
- 已关闭节点颜色更淡
- 失败节点在树和主图中都必须有一致的错误语义

### 4.2.5 节点详情区

详情区用于展示选中对象的完整上下文。

建议分成以下卡片块：

1. 基本信息
2. 状态信息
3. 关联对象
4. 最近事件
5. 日志摘要
6. 指标摘要

#### 基本信息

显示：

- 对象 ID
- 对象类型
- 父级对象
- 所属 run
- 创建 / 更新时间

#### 状态信息

显示：

- 当前状态
- 当前阶段
- 最近一次状态变更时间
- 当前是否活跃
- 是否已关闭

#### 关联对象

显示：

- 上游关联
- 下游关联
- 同层关联
- 相关 worker / episode / step

#### 最近事件

显示最近 5 到 10 条事件，按时间倒序：

- 事件类型
- 事件时间
- 简短摘要
- 可跳转到原始事件

#### 日志摘要

只显示筛选后的关键日志，不要把完整日志流塞进详情面板。

#### 指标摘要

显示最关键指标，例如：

- 成功率
- 失败率
- 最近耗时
- 重试次数
- 当前积压量

## 5. 底部辅助区设计

底部辅助区用于“辅助判读”，不应抢占主注意力。

### 5.1 区块划分

建议以标签切换方式组织：

- 日志
- 指标
- 事件流
- 快照列表
- 搜索结果

### 5.2 日志区

日志区建议支持：

- 按对象筛选
- 按级别筛选
- 按时间倒序
- 关键词搜索

#### 日志行的组成

- 时间
- 级别
- 来源对象
- 简短消息
- 展开后的原始字段

### 5.3 指标区

指标区建议用卡片 + 小型趋势列表的方式。

建议展示的指标包括：

- 当前吞吐
- 成功 / 失败计数
- 平均耗时
- 最近 5 分钟波动

### 5.4 事件流区

事件流区用于显示系统级异步事件，强调时序。

建议每条事件展示：

- 序号
- 时间
- 事件类型
- 来源模块
- 关联对象

### 5.5 快照列表

快照列表应支持：

- 按时间排列
- 标记抓拍来源
- 一键切换到快照
- 删除或重命名本地快照

---

## 6. 核心视觉细节

### 6.1 颜色分工

颜色不只是状态，还要承担视觉分层任务。

建议建立三类颜色使用：

- 状态色：成功、失败、进行中、关闭
- 层级色：主区、辅区、背景、边界
- 提示色：告警、定位、强调

### 6.2 字体和信息呈现

不同信息类型应该有明确视觉权重：

- 页面标题最大
- 节点名称次之
- 说明文本再次之
- ID、时间、事件序号使用等宽字体

### 6.3 卡片与分隔

卡片不是为了堆叠，而是为了分组。

规则建议：

- 每张卡片只表达一个主题
- 同类信息放同一层级
- 卡片间留出足够阅读间距
- 主图区域不要被大量卡片切碎

### 6.4 选中态和聚焦态

选中态必须在所有联动区域保持一致：

- 主图节点选中
- 树状图节点选中
- 详情面板联动
- 底部日志同步筛选

聚焦态和选中态不一样：

- 聚焦是“当前重点观察对象”
- 选中是“当前操作对象”

---

## 7. 页面状态设计

### 7.1 实时状态

实时状态下页面持续接收更新。

表现方式：

- 活跃节点微弱高亮
- 最新事件浮现到事件流顶部
- 右上角显示连接状态
- 更新时间持续刷新

### 7.2 快照状态

快照状态下页面数据固定。

表现方式：

- 顶部显示快照时间
- 动态更新标识关闭
- 主图和树图保持冻结
- 底部日志按快照时间点回溯

### 7.3 断连状态

如果 SSE 或数据流断开：

- 顶部显示断连提示
- 主图保留最后状态
- 提供重连按钮
- 不清空当前屏幕内容

### 7.4 运行结束状态

run 结束后：

- 主图保留最终状态
- 活跃标记消失
- 终态节点强调
- 快照功能保持可用

## 8. 前后端对接功能清单

这一部分是页面真正需要对接的后端能力，不是抽象概念。

### 8.1 Run 级控制接口

#### 需要的后端功能

- 创建 run
- 终止 run
- 查询 run 当前状态
- 查询 run 运行摘要
- 获取 run 历史快照列表

#### 页面用途

- 顶部开始 / 停止按钮
- 状态条展示
- 实时摘要刷新
- 快照模式切换入口

### 8.2 实时流接口

#### 需要的后端功能

- SSE 或同等单向事件推送
- 全量状态初始化
- 增量状态更新
- 运行状态变更事件
- 心跳保活

#### 页面用途

- 工作流视图实时刷新
- 树状图同步刷新
- 详情面板自动更新
- 事件流追加

### 8.3 对象详情接口

#### 需要的后端功能

- 按 `training_run_id` 查询对象树
- 按对象类型和对象 ID 查询详情
- 按对象查询最近事件
- 按对象查询关联日志
- 按对象查询指标摘要

#### 页面用途

- 点击树节点后加载详情
- 点击工作流节点后联动右侧面板
- 展示对象历史上下文

### 8.4 搜索接口

#### 需要的后端功能

- 按 ID 搜索 run / worker / episode / step
- 按关键词搜索事件或日志
- 按状态筛选对象

#### 页面用途

- 顶部搜索框
- 底部搜索结果面板
- 快速跳转到对应节点

### 8.5 日志接口

#### 需要的后端功能

- 按 run / 对象 / 时间范围拉取日志
- 按 level 过滤
- 分页或增量加载

#### 页面用途

- 底部日志区
- 详情面板中的日志摘要
- 快速定位异常来源

### 8.6 指标接口

#### 需要的后端功能

- 拉取 run 级指标
- 拉取对象级指标
- 查询短时间窗口趋势

#### 页面用途

- 底部指标区
- 详情卡片中的关键指标
- 运行健康度摘要

### 8.7 快照接口

#### 需要的后端功能

- 创建快照
- 查询快照列表
- 获取快照内容
- 删除或重命名快照

#### 页面用途

- 顶部抓拍按钮
- 底部快照列表
- 快照模式切换

## 10. 典型交互流程

### 10.1 查看当前运行

1. 用户进入页面
2. 顶部显示当前 run 和连接状态
3. 左侧工作流图呈现当前进度
4. 右侧树状图展开到活跃层级
5. 详情区显示当前选中节点信息

### 10.2 定位异常节点

1. 用户看到某个节点异常高亮
2. 点击该节点
3. 右侧详情区显示原因与关联对象
4. 底部日志过滤到对应对象
5. 用户根据最近事件进一步追踪

### 10.3 抓取快照并回看

1. 用户点击抓拍
2. 后端返回快照 ID 或时间戳
3. 页面保存当前状态副本
4. 用户切换到快照模式
5. 所有视图进入冻结态展示

### 10.4 搜索对象

1. 用户在顶部输入 ID 或关键词
2. 页面返回匹配结果
3. 结果可定位到工作流节点或树节点
4. 右侧详情更新

## 11. 布局在不同窗口下的变化

### 11.1 宽屏

宽屏下保持：

- 左侧主图完整
- 右侧树和详情同屏
- 底部辅助区横向展开

### 11.2 中屏

中屏下：

- 右侧详情缩短
- 树状图保留，详情可折叠
- 底部区域改成标签页

### 11.3 窄屏

窄屏下：

- 主图优先
- 右侧改抽屉
- 底部辅助区改为单列

---

## 12. 设计验收标准

这套 UI 设计完成后，应当满足以下标准：

1. 用户一眼能分清实时、快照和结束状态
2. 用户能从主图看出当前推进到哪个阶段
3. 用户能通过树状图快速下钻到对象层级
4. 用户能在详情区看到选中对象的完整上下文
5. 用户能通过底部日志和指标确认异常原因
6. 用户能明确知道页面要对接哪些后端功能

如果页面做完后仍然出现以下问题，就说明设计不够落地：

- 主图和树图职责重叠
- 详情面板放太多原始字段
- 日志区和主图争夺注意力
- 快照和实时状态没有明显区分
- 运行状态、对象状态、事件状态混在一起

---

## 13. 结论

这份设计的核心，不是“把信息摆上去”，而是明确每个区域承担什么职责：

- 顶部负责状态和控制
- 左侧负责看整体推进
- 右侧负责看层级和细节
- 底部负责辅助分析

同时，前后端对接也要按这个职责划分，分别提供运行控制、实时流、对象详情、日志、指标和快照能力。

如果后续继续细化，下一步应该直接拆成两份补充文档：

1. 页面线框图级别的分区草图说明
2. 后端接口与前端状态映射表
