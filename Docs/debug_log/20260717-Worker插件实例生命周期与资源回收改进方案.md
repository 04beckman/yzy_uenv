# Worker 插件实例生命周期与资源回收改进方案

- 日期：2026-07-17
- 范围：`uenv-worker` 插件（`math` / `code`）进程级实例的生命周期管理与资源回收
- 背景：2026-07-17 运维中在 7143 发现 **671 个 `uenv-math-plugin` + 55 个 `uenv-code-plugin`** 孤儿进程，本方案分析其成因并给出一次性改进规划。
- 关联：[Math与DSCodeBench错误修复与Worker联调记录.md](20260717-Math与DSCodeBench错误修复与Worker联调记录.md)

## 1. 现状分析

### 1.1 已有的生命周期管理（账面层：较完整）

`uenv-worker/src/pool/warmup_pool.rs` 已实现较完整的实例状态机与策略：

- 状态：`Creating / Warm / Active / Idle / Cooling / Evicting / Destroyed`；
- `acquire` / `release`：含 double-allocation 检测、复用前 `health_check`、归还时 `reset`；
- 回收策略（`config/uenv-worker.deploy-7143-swe-pro.yaml`）：
  - `warmup_size=4`、`prewarm_on_startup=true`
  - `max_idle_time=300`、`cool_timeout=60`
  - `max_episode_count=1000`
- `evict_idle` 按空闲时长驱逐、`release` 按 episode 计数触发回收，`destroy_instance` 负责下线并从 tracked / warm 队列移除。

### 1.2 缺陷（物理进程层：基本缺失）

账面记账正确（`destroy_instance` 会标记 `Evicting`、移出队列与表），但**底层操作系统进程从未被真正终止**。存在三处叠加缺陷：

**缺陷 A — `close()` 的 kill 分支是死代码。**
`PluginHost::spawn` 后立即启动 exit watcher，将 `Child` 句柄从 `ManagedInstance` 中 `take` 走：

```270:289:uenv-worker/src/plugin/host.rs
    fn spawn_exit_watcher(&self, instance_id: String) {
        ...
                if let Some(managed) = guard.instances.get_mut(&instance_id) {
                    managed.child.take()   // spawn 后句柄即被移走
                } else { None }
            ...
            if let Some(mut child) = maybe_child {
                let _ = child.wait().await;   // 插件永不自退 → 永久阻塞
```

因此 `close()` 执行时 `managed.child` 恒为 `None`，`kill().await` 永不触发：

```209:229:uenv-worker/src/plugin/host.rs
    pub async fn close(&self, instance_id: &str) -> Result<(), ...> {
        let managed = { ... state.instances.remove(instance_id) };
        ...
            if let Ok(mut client) = PluginRpcClient::connect_uds(...).await {
                let _ = client.close().await;    // no-op gRPC，插件不退出
            }
        if let Some(mut child) = managed.child.take() {  // 恒为 None
            let _ = child.kill().await;                  // 永不执行
        }
        let _ = fs::remove_file(&managed.metadata.uds_path);  // 仅删 socket
```

`close()` 净效果：发一个 no-op gRPC close、删 socket 文件、移除记录——**进程仍存活**。

**缺陷 B — 插件自身 `close` 处理器不自退。**
`uenv-math-plugin` / `uenv-code-plugin` 的 `close` 仅返回 `ok=true`，不关闭 gRPC server、不 `exit`；收到 close 后进程继续运行，缺乏“优雅下线”通路。

**缺陷 C — 无 `kill_on_drop`，且关停不清理。**
`ProcessBackend::create` 未设 `.kill_on_drop(true)`：

```31:39:uenv-worker/src/backend/process.rs
        let child = cmd
            .arg("--uds-path").arg(uds_path)
            .stdin(Stdio::null())
            .stdout(Stdio::null())
            .stderr(Stdio::inherit())
            .spawn()?;   // 未设 kill_on_drop(true)
```

Worker 优雅关停（`serve_with_shutdown` 返回）后只打印 `worker_stop`，**未遍历 pool 做 close/kill**：

```351:359:uenv-worker/src/runtime.rs
            .serve_with_shutdown(addr, shutdown_signal())
            .await?;
        tracing::info!(..., msg = "worker_stop");
        Ok(())   // 无插件池清理
```

### 1.3 叠加效果与证据

- **运行期泄漏**：每次 `evict_idle` / `release`(reset|health 失败) / `max_episode_count` 回收 → `destroy_instance → close()` → 泄漏一个存活进程。
- **跨重启累积**：`pkill uenv-worker` 后插件被 init（pid 1）收养继续存活，下次重启又起一批；孤儿 socket 名带**多个历史 worker PID**（`uenv-math-3017669-*`、`uenv-math-539536-*`、`uenv-code-2663930-*`…），证明是多代 Worker 遗留累积。
- **性质澄清**：exit watcher 的 `child.wait()` 会回收真正退出的子进程，故不产生 `<defunct>` 僵尸；问题在于**无人令插件退出**，它们是**存活的孤儿进程**，占用 RSS 与 PID。

### 1.4 缺口小结

| 层面 | 现状 |
|---|---|
| 实例状态机 / 记账 | 有，较完整 |
| 空闲驱逐 / episode 回收策略 | 有 |
| **进程终止（close 真正 kill）** | **缺失（死代码）** |
| **插件优雅下线（自退）** | **缺失** |
| **关停时批量回收** | **缺失** |
| **孤儿巡检 / 对账清理** | **缺失** |

### 1.5 关联发现：Server 侧重试策略缺陷（独立主题）

本轮排查中另外发现一个**属于 Server 侧、与进程回收不同主题**的问题，一并记录：

Server 把 Worker 返回的**确定性执行失败**（如 math panic 对应的 `ERR_ENV_STEP_FAILED`）统一包装成可重试的 `dispatch_failed`，按 `episode.max_attempts` 盲目重试，每次都重复昂贵的模型生成：

```294:389:uenv-server/src/service/episode.rs
                            worker_lease.release();
                            Some(format!("dispatch_failed: {e}"))
                        ...
            if let Some(reason) = retry_reason {
                if reason.contains("max_concurrency_acquire_timeout") {
                    // 只有 worker 并发满才不消耗 attempt
                ...
                } else {
                    req.attempt_id += 1;   // 其余一律重试，含确定性失败
```

分类函数 `is_retryable_schedule_error`（`uenv-server/src/service/prelude_and_guards.rs`）只作用于“选 worker”阶段（无 worker/满载可重试、env 不匹配不重试），**并不作用于执行结果失败**。因此所有执行失败都会被重试到 `max_attempts`。这正是 [数学错误排查.md](../worker/260717/数学错误排查.md) 点名的“次要缺陷”：149 个确定性失败额外产生约 298 次模型请求。

判定与边界：

- **与进程回收/僵尸无关**，是另一个主题（避免重复模型算力浪费），因此**不纳入本方案的 Worker 侧实现**；
- Worker 侧 panic 根因已修复后，这类确定性失败当前不再被触发，问题已不发作；
- 作为健壮性改进，Server 侧需要将确定性 step 失败（`ERR_ENV_STEP_FAILED`）标记为不可重试、直接进入终态，避免重复模型生成。

**2026-07-19 状态更新：已完成。** 当前 `uenv-server/src/service/episode.rs` 在 dispatch 返回错误后调用确定性错误分类；命中 `ERR_ENV_STEP_FAILED` 时立即构造失败终态，写入 `failure_source=worker_execution_failed` 与 `retryable=false`，不再消耗后续 attempt。`uenv-server/src/service/tests.rs::explicit_env_step_failure_is_non_retryable` 覆盖显式错误码，兼容旧 Worker 的 h2 reset/panic 特征也有单独测试。本缺陷不再是待排期项。

## 2. 改进规划（一次性完成，按模块分工）

目标：使“账面下线”与“物理进程终止”严格一致，并提供关停清理与孤儿兜底，确保任何回收路径都不再泄漏进程。

### 2.1 `uenv-worker/src/backend/process.rs`（进程后端）

- `ProcessBackend::create` 增加 `.kill_on_drop(true)`：作为最后一道保险，`Child` 句柄被 drop 时由 tokio 发送 kill，避免任何遗漏路径导致的泄漏。
- 保持 `stderr(Stdio::inherit())`（已在前序修复中加入，用于保留插件 panic 线索）。

### 2.2 `uenv-worker/src/plugin/host.rs`（实例宿主，核心）

- **修正句柄归属**：不再在 `spawn_exit_watcher` 中 `take` 走 `Child`。将进程句柄保留在 `ManagedInstance`（或改为保存 `pid` + 关停信号），使 `close()` 能够真正拿到并终止进程。
- **`close()` 真正终止进程**：
  1. 先发 gRPC `close` 作为优雅信号（best-effort，带超时）；
  2. 随后 `child.start_kill()` / `kill().await` 强制终止，再 `wait().await` 回收，杜绝残留；
  3. 删除 `uds_path` 及其 `*.episode.json` 伴随文件。
- **exit watcher 重构**：改为“只观察、不夺取句柄”——用于在进程异常退出时把实例标记 `Broken` 并从池中剔除；与 `close()` 的主动终止不冲突（避免双重 `wait` 竞争，需以状态/一次性通道协调）。
- **新增批量关停接口**：`close_all()`，遍历所有实例执行上面的终止逻辑，供关停与巡检复用。
- **新增启动巡检**：`reap_orphans()`——按“非当前进程 PID 归属”清理 `/tmp/uenv-*.sock`、`*.episode.json` 陈旧文件（进程终止交给下一条的运维/脚本或 pattern kill），把本次手动清理固化为代码能力。

### 2.3 `uenv-worker/src/pool/warmup_pool.rs`（实例池）

- `destroy_instance` 依赖 2.2 修正后的 `close()`，保证状态迁移与物理终止一致；补充失败日志（close 失败时 WARN，便于观测）。
- 暴露 `shutdown()`：将池内 tracked/warm/active 全部经 `PluginHost::close` 下线（内部转调 `close_all` 或逐个 close），供 runtime 关停调用。
- 可选：`status_counts` 增加 “leaked/所有权异常” 观测项（用于 metrics 校验记账与实际进程数一致）。

### 2.4 `uenv-worker/src/runtime.rs`（运行时编排）

- `serve_with_shutdown` 返回后、打印 `worker_stop` 前，调用 `WarmupPool::shutdown()` 完成插件池批量回收，确保 SIGTERM/Ctrl-C 优雅关停不留孤儿。
- 启动阶段（prewarm 之前）调用 `PluginHost::reap_orphans()`，清理上一代 Worker 遗留的陈旧 socket / 进程。

### 2.5 插件二进制 `uenv-worker/src/bin/uenv-math-plugin.rs` / `uenv-code-plugin.rs`

- `close` 处理器改为触发优雅下线：返回响应后驱动 gRPC server 停止（如通过 `serve_with_shutdown` + 关停通道），进程随之 `exit(0)`，使 gRPC `close` 成为正常下线通路，与 host 侧的强制 kill 形成“优雅优先、强制兜底”的两级机制。

### 2.6 观测与自检（跨模块）

- `28777` metrics 增加插件实例计数（warm/active/tracked）与实际子进程数对账指标；
- 关停日志补充 `pool_shutdown_closed=<n>`、启动日志补充 `reaped_orphans=<n>`，便于回归验证。

## 3. 验收标准

1. 单测：`close()` 后目标 PID 不再存在；`shutdown()` / `close_all()` 后 tracked 为空且无对应子进程存活。
2. 集成：一次 acquire→release→evict/recycle 循环后，`pgrep -f uenv-*-plugin` 计数回到 `warmup_size` 量级，不随 episode 数单调增长。
3. 关停：`SIGTERM` 后无遗留 `uenv-*-plugin`；重启后无跨代累积。
4. 启动巡检：存在陈旧 socket / 上一代进程时，启动日志 `reaped_orphans>0` 且清理生效。

---

## 4. 风险与回滚

- **双重 wait 竞争**：exit watcher 与 `close()` 均可能 `wait` 同一 child，需以一次性通道 / 状态标志协调，避免 panic 或句柄争用（重点测试项）。
- **关停顺序**：须先停 gRPC 入口再回收池，避免关停期间仍有新 `acquire` 进入。
- **kill_on_drop 副作用**：确保正常复用路径不会因句柄意外 drop 而误杀 warm 实例（句柄始终随 `ManagedInstance` 生命周期持有）。
- 回滚：各模块改动相互独立，可按文件维度 git 反向 patch；最小闭环为 2.1 + 2.2 + 2.4，可单独回退观测项与插件自退部分。

## 5. 实施结果（2026-07-17 落地）

规划的所有 Worker 侧修复已实现并完成本地 + 服务器联调，服务已重启，运行正常。

### 5.1 代码改动明细

| 模块 | 改动 | 对应规划 |
| --- | --- | --- |
| `uenv-worker/src/backend/process.rs` | 子进程 spawn 增加 `.kill_on_drop(true)`，Child 句柄 drop 即 kill，作为最后兜底 | 2.1 |
| `uenv-worker/src/plugin/host.rs` | 移除“夺取 Child 句柄”的 `spawn_exit_watcher`，句柄留在 `ManagedInstance`；`close()` 改为「gRPC close(带 2s 超时) → `start_kill`+`wait` 强制回收 → 清理 socket/episode.json」真正终止进程；新增 `close_all()` 批量下线；新增 `reap_orphans()` 启动巡检（Linux 扫 `/proc` kill 遗留 `uenv-*-plugin`，跨 unix 清理陈旧 `uenv-*.sock*`） | 2.2 |
| `uenv-worker/src/pool/warmup_pool.rs` | `destroy_instance` 在 `close` 失败时打 WARN 日志；新增 `shutdown()` 将 tracked 实例全部经 `destroy_instance` 回收 | 2.3 |
| `uenv-worker/src/runtime.rs` | 启动 prewarm 前调用 `PluginHost::reap_orphans()`（`reaped_orphans>0` 打 WARN）；`serve_with_shutdown` 返回后调用 `WarmupPool::shutdown()` + `PluginHost::close_all()`，`worker_stop` 日志补充 `pool_shutdown_closed`/`host_closed` | 2.4 / 2.6 |
| `uenv-worker/src/bin/uenv-math-plugin.rs`、`uenv-code-plugin.rs` | `close` 处理器返回响应并触发关停通道，`serve_with_incoming_shutdown` 收到信号后进程优雅 `exit(0)`，形成「优雅优先、强制兜底」两级下线 | 2.5 |
| `uenv-worker/tests/m4_plugin_host_process.rs` | 新增回归测试 `m4_close_terminates_plugin_process`：`close()` 后目标 PID 必须不存在 | 验收 1 |

> 设计取舍：原 `spawn_exit_watcher` 会在 spawn 时夺取 Child 句柄，导致 `close()`/`terminate_for_test()` 永远拿不到句柄、从不真正 kill（孤儿根因之一，同时是「双重 wait 竞争」的来源）。本次直接移除该 watcher，句柄始终随 `ManagedInstance` 持有；异常退出实例改由复用前的 `health_check` 检出并销毁，从根上消除双重 wait 竞争。

### 5.2 验证结果

- **本地**：`cargo build -p uenv-worker --bins` 通过（无 warning）；`m4_plugin_host_process` 全部 3 项通过（含新增 `close` 杀进程回归）。`--lib` 中 `model_client::uses_typed_generation_config_json` 偶发失败为既有并发 flake（单独运行通过），与本次改动无关。
- **服务器（worker-7143-pro / 219.147.100.43:7143）**：`cargo build --release` 15s 通过；`m4_plugin_host_process` 3 项通过。
- **联调 —— 启动巡检**：以新二进制重启时，旧（无关停回收）Worker 停止后遗留 8 个插件孤儿，新 Worker 启动日志 `reaped_orphans=8 msg="reaped_orphan_plugins_on_startup"`，随后预热稳定在 `math=4 code=4`，全部插件父进程为新 Worker。
- **联调 —— 优雅关停**：对新 Worker 发送 `SIGTERM`，日志 `worker_stop pool_shutdown_closed=8 host_closed=0`（池已回收 8 个实例并从 host 移除，故 host_closed 为 0），关停后 `worker=0 math=0 code=0`，**零孤儿残留**。
- **联调 —— 稳态**：再次重启，因上次为优雅关停无遗留，启动巡检无需清理（无 `reaped_orphans` 日志）；`health=ok`、`worker=1 math=4 code=4`、register/gateway 正常；无 panic/`ERR_`（仅既有的 hub swe 版本 404 本地回退 WARN，与本次无关）。

### 5.3 结论

孤儿插件进程的三条根因（`close()` 从不 kill、缺少关停批量回收、无启动巡检 + 无 `kill_on_drop` 兜底）已全部闭合：正常复用/回收即时杀进程、优雅关停零残留、跨代重启由启动巡检兜底清理。第 1.5 节的 Server 侧确定性失败重试策略也已于后续提交闭合；Worker 生命周期与该 Server 重试缺陷均不再是待实现项。
